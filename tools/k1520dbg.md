# k1520dbg — interaktiver Debugger für die A5120/K1520-Emulation

`k1520dbg` ist ein gdb-artiger Kommandozeilen-Debugger um `A5120Machine` (den neuen
K1520-Core). Er debuggt **beide Z80** der ZRE/K2526 gleichzeitig — die Haupt-CPU
**ZVE1** *und* den DMA-Prozessor **ZVE2** — sowie jedes geladene Programm
(Bootloader, CP/A, `format.com`, …), das nativ auf der Maschine läuft.

> **Warum nicht boot_trace?** `boot_trace` filtert nach **absoluten** Zyklen ab
> Power-On — ideal für die Boot-ROM-Phase, aber unhandlich, sobald ein geladenes
> Programm läuft (man muss erst die absolute Zyklenzahl ermitteln und neu starten).
> `k1520dbg` arbeitet statt dessen mit **Breakpoints**, **Single-Step**, einem
> **relativen Zyklen-Marker** und einer interaktiven Sitzung. Faustregel:
> *boot_trace zum Lokalisieren der groben Phase, k1520dbg zum Sezieren.*

---

## 1. Bauen & Starten

```sh
cmake --build build --target k1520dbg -j

# interaktiv:
./build/k1520dbg disks/cpadisk01.img

# Skript-/Pipe-getrieben (für Regression/Reproduktion):
printf 'b 0xC7A3\ng\nr\nq\n' | ./build/k1520dbg disks/cpadisk01.img
./build/k1520dbg disks/cpadisk_02.hfe -x mein_skript.dbg
```

Aufruf:

```
k1520dbg [DISK] [-x SCRIPT] [-s SYMFILE] [-l LISTING.prn]
  DISK        .img oder .hfe; wird auf A: gemountet (Format cpa780/cpa800 autodetektiert)
  -x SCRIPT   Kommandos aus Datei abarbeiten, dann auf stdin wechseln (wiederholbar nein)
  -s SYMFILE  Symboltabelle laden (s. §6) — wiederholbar
  -l LISTING  MACRO-80-.prn-Listing laden → Disasm mit Original-Quelle+Kommentar
              annotieren (s. §6a) — wiederholbar
```

Beispiel mit BIOS-Listing:

```sh
./build/k1520dbg -l ~/projects/CPA_Workbench/build/bios.prn disks/cpadisk01.img
```

Kommandos kommen aus dem `-x`-Skript (zuerst) und danach von stdin. `#`-Zeilen sind
Kommentare. `q`/`quit` beendet, EOF auf stdin ebenfalls.

## 2. Grundkonzepte

**Zwei CPUs.** ZVE1 ist die Haupt-CPU; ZVE2 ist der DMA-Prozessor, der nur läuft,
während `/BUSRQ` gehalten wird (Bootsektor-/Floppy-Transfers). Befehle ohne Suffix
betreffen ZVE1; ZVE2-Varianten enden auf `2` (`b2`, `s2`, `set 2`). `r 2` und die
Statuszeile zeigen den ZVE2-Zustand (`reset`/`wait`/`run`, `BUSRQ`).

**Break-before-execute.** Der Trace-Callback feuert **vor** der Instruktion. An einem
Breakpoint steht der PC also *auf* dem noch nicht ausgeführten Befehl, und `SP` zeigt
auf die Rücksprungadresse. `=>` markiert in jeder Ausgabe die nächste Instruktion.

**Zeitbasis.** `cyc`/`c-cyc` sind ZVE1-Taktzyklen (instruktionsgenau, monoton ab
Power-On). Mit `mark` (§7) setzt man einen relativen Nullpunkt; danach zeigen alle
Ausgaben `+<relzyklen>`.

**Eingebauter Disassembler.** Jeder Stopp, jeder Step und `u` disassemblieren über
`tools/z80dis_min.h` (vollständige Z80-Tabelle inkl. ED/CB/DD/FD, gegen das
Boot-ROM byte-/längengenau verifiziert). Kein externer Python-Aufruf mehr nötig.

## 3. Zahlen- und Ausdruckssyntax

| Kontext | Default-Basis | Beispiele |
|---------|---------------|-----------|
| **Adressen / Zähler** (`b`, `gu`, `d`, `u`, `g N`, `s N`, …) | dezimal (base-0) | `0xC7A3`, `0C7A3H`, `49827`, `STAGE1`, `SELDSK+5` |
| **Poke-Bytes** (`e`) | **hex** | `e 0x6000 CD 10 60` schreibt CD 10 60 |

→ Adressen/Zähler **immer mit `0x…` oder `…H`** schreiben, wenn hex gemeint ist
(`d 6000` wäre dezimal 6000 = 0x1770!). Symbolnamen (§6) sind überall erlaubt, wo
eine Adresse erwartet wird, auch mit Offset (`name+0x10`).

**Ausdrücke** (für `b … if`, `disp`):

```
REG          A F B C D E H L  AF BC DE HL IX IY SP PC  I R   (groß/klein egal)
(rr)         (HL) (DE) (BC) (SP)        — Byte an der Pair-Adresse
[addr]       Byte im Speicher           — addr = Zahl oder Symbol
[addr]w      16-Bit-Wort (LE) im Speicher
zahl/symbol  Konstante
```

**Bedingung:** `LHS OP RHS` mit `OP ∈ { == != < > <= >= }`, z. B.
`A==0xA9`, `[03F8]!=3`, `[D1BE]w==0`, `B>5`.

## 4. Kommandoreferenz

### Ablaufsteuerung
| Befehl | Wirkung |
|--------|---------|
| `g` / `c` `[N]` | weiterlaufen bis Breakpoint (oder genau `N` ZVE1-Zyklen) |
| `gu <A>` | bis ZVE1 die Adresse `A` erreicht (einmaliger Temp-Breakpoint) |
| `s [N]` | `N` ZVE1-Instruktionen **hinein**-steppen (Default 1), jede getrace-disassembliert |
| `s2 [N]` | `N` **ZVE2**-Instruktionen steppen (nur wirksam während `/BUSRQ`/DMA) |
| `n [N]` | `N` ZVE1-Instruktionen **über**-steppen — `CALL`/`RST` und repetierende ED-Blockbefehle (`LDIR`,`INIR`,…) werden bis zum Folgebefehl durchlaufen |
| `fin` | aus der aktuellen Funktion **heraus**-laufen (bis `SP` über den aktuellen Rahmen steigt) |

### Reverse-Debugging & Snapshots
| Befehl | Wirkung |
|--------|---------|
| `rs [N]` | **Reverse-Step**: die letzten `N` Vorwärts-Kommandos rückgängig machen (auto-Snapshot-Ring, §9) |
| `snap <name>` | benannten Voll-Snapshot speichern |
| `restore <name>` | benannten Snapshot wiederherstellen |
| `snap list` | benannte Snapshots auflisten |

### Breakpoints
| Befehl | Wirkung |
|--------|---------|
| `b <A> [if <cond>]` | Breakpoint auf **ZVE1** (optional bedingt, §3) |
| `b2 <A> [if <cond>]` | Breakpoint auf **ZVE2** |
| `tb <A>` | einmaliger (selbstlöschender) Breakpoint auf ZVE1 |
| `bd <A>` / `bd2 <A>` | Breakpoint löschen (ZVE1 / ZVE2) |
| `be <A>` / `bdis <A>` | Breakpoint **aktivieren** / **deaktivieren** (behalten, aber inaktiv); ZVE2: `be2`/`bdis2` |
| `bi <A> <N>` | die nächsten `N` Treffer **ignorieren**, dann anhalten (gdb `ignore`); ZVE2: `bi2` |
| `bl` | alle Breakpoints listen — mit Trefferzähler, `[disabled]`, `ignore=N`, Bedingung und Symbol |

### Watchpoints (Speicher & I/O)
| Befehl | Wirkung |
|--------|---------|
| `wp <A\|A..B> [cond]` | **drucken** bei Schreibzugriff (Einzeladresse oder Bereich) |
| `wpr <A\|A..B> [cond]` | drucken bei **Lese**zugriff |
| `wb <A\|A..B> [cond]` | **anhalten** bei Schreibzugriff (Daten-Breakpoint) |
| `wd <A>` / `wd all` / `wl` | Watch löschen (alle, die `A` abdecken / alle) / listen (mit Trefferzähler) |
| `iow <P>` | drucken bei jedem I/O-Zugriff auf Port `P` (IN/OUT + Daten + PC) |
| `iob <P>` | anhalten bei I/O-Zugriff auf Port `P` |
| `iod <P>` / `iol` | I/O-Watch löschen / listen |

**Bereich:** `A..B` überwacht den ganzen Adressbereich (z. B. `wp 0xD1B0..0xD1CF`).
**Wert-Bedingung** `[cond]` (optional, nur bei Schreib-/Lese-Wert):
`== NN` / `!= NN` (NN ist ein Byte, **hex** wie bei `e`/`d`) oder `changed`
(nur feuern, wenn sich der Wert gegenüber dem letzten Schreiben *tatsächlich* ändert).
Beispiele: `wb 0x03F8 == 3`, `wp 0xD1BE != 00`, `wp 0x6000..0x60FF changed`.

> Jeder Treffer meldet die **tatsächlich auslösende CPU** (`ZVE1`/`ZVE2.PC=…`) — während
> einer DMA ist das ZVE2, sonst ZVE1.

### Logpoints & Trace-to-File (laufen lassen **und** mitschreiben)
| Befehl | Wirkung |
|--------|---------|
| `logpoint <A> [expr…]` (`lp`) | am PC `A` **drucken und weiterlaufen** (gdb `dprintf`): PC + Disasm + ausgewertete Ausdrücke (§3), nie anhalten |
| `lpd <A>` / `lpl` | Logpoint löschen / listen |
| `trace <file> [lo hi]` | **jede** ausgeführte Instruktion (ZVE1 *und* ZVE2) disassembliert + `.prn`-annotiert in `file` schreiben; optional nur im PC-Fenster `[lo,hi]` |
| `trace off` | Trace beenden, Datei schließen, Zeilenzahl melden |

> `trace` schreibt lückenlos über DMA-Phasen (ZVE2 wird mitgeschrieben). Sicherheits-Cap
> 2 000 000 Zeilen (dann Stopp + Hinweis) — Fenster `lo hi` oder `g <N>` begrenzen den Umfang.
> Ersetzt das „von Hand steppen": einmal `trace`, einmal `g`, und der komplette,
> kommentierte Lauf liegt in der Datei.

### Inspektion
| Befehl | Wirkung |
|--------|---------|
| `r [2]` | ZVE1-Register (+Flags `SZHPNC`, Schattenregister, `->`Rücksprung); `r 2` zusätzlich ZVE2 |
| `bt [N]` | **exakter** Backtrace aus dem mitlaufenden CALL/RST/RET-Aufrufstapel (§9); `bt scan [N]` erzwingt den alten heuristischen Stack-Scan |
| `d <A> [N]` | Hex+ASCII-Dump (Default 64 Byte) |
| `u [A] [N]` | `N` Instruktionen ab `A` disassemblieren (Default: ab aktuellem PC, 12 Instr.; ohne `A` setzt `u` fort) |
| `x/<N><fmt><sz> <A>` | gdb-artiges **Examine** (s. u.); ohne `<A>` läuft `x` ab der letzten Adresse weiter |
| `list` / `l [A] [N]` | **`.prn`-Quelltext** um `A` (Default PC / weiter); markiert die deckende Zeile mit `=>` — braucht ein geladenes Listing (§6a) |
| `e <A> <b…>` | Bytes poken (**hex**), z. B. `e 0x6000 3E 00 C9` |

**`x` — Speicher in beliebigem Format ansehen** (`x/<count><fmt><size> <addr>`):

| | Bedeutung |
|--|--|
| `count` | Anzahl Elemente (Default 1 bzw. vom letzten `x`) |
| `fmt` | `x` hex · `d` signed-dez · `u` unsigned-dez · `c` Zeichen · `t` binär · `o` oktal · `a` Adresse (+Symbol) · `i` Instruktion (disasm + `.prn`) · `s` NUL-String |
| `size` | `b` Byte · `w`/`h` Wort (2 Byte, little-endian) — bei `i`/`s` ignoriert |

`fmt`, `size` und `count` **bleiben erhalten** (wie in gdb), bis sie neu angegeben werden.
Jede Zeile zeigt die Adresse (+Symbol, falls vorhanden). Beispiele:
`x/8xb 0xD1B0` · `x/4dw HL` · `x/10i 0x1E40` · `x/s 0xF800` · `x/2a 0xD1B8` · `x` (weiter).
| `set [2] <reg> <v>` | Register setzen (ZVE1, mit `2` ZVE2): `set HL 0x6000`, `set 2 A 5` |
| `disp <expr>` | Ausdruck zur **Display-Liste** hinzufügen (wird an jedem Stopp ausgewertet) |
| `disp` / `undisp <n>` | Display-Liste anzeigen / Eintrag `n` entfernen |
| `vars` | benannte CP/A-RAM-Variablen (`[03F8]`-Done-Flag, DPB-Felder) |
| `dev` | K5122-Controller-Zustand: gewähltes Laufwerk, **Zylinder/Kopf**, Transfer aktiv (`READING`), `headPos`/Spurlänge, Sektorgröße, `/BUSRQ`-Pending |

### Speicher laden/sichern
| Befehl | Wirkung |
|--------|---------|
| `load <f> <A>` | Binärdatei `f` ab Adresse `A` in den RAM schreiben (Patch/Programm injizieren) |
| `save <f> <A> <N>` | `N` Byte ab `A` in Datei `f` sichern (z. B. für externe Disassemblierung) |

### Symbole & Sonstiges
| Befehl | Wirkung |
|--------|---------|
| `sym <f>` | Symboldatei laden (§6) |
| `sym add <name> <A>` | einzelnes Symbol setzen |
| `sym list` | geladene Symbole listen |
| `lst <f.prn>` | MACRO-80-.prn-Listing laden → Disasm annotieren (§6a) |
| `lst list` | geladene Listing-Zeilen (Adresse → Quelle) ausgeben |
| `mark [A]` | relativen Zyklenzähler **jetzt** nullen, oder bei Erreichen von `A` scharfschalten (§7) |
| `keys <text>` | Text tippen und die Maschine dabei weiterlaufen lassen (`\r`=Enter, `\t`=Tab, `\e`=ESC) |
| `screen` | 80×24-Text-VRAM (ab `0xF800`) ausgeben |
| `reset` | Power-Cycle (Boot-ROM wieder aktiv) |
| `q` / `quit` | beenden |

## 5. Workflow: ein geladenes Programm debuggen (z. B. `format.com`)

Geladene Programme laufen nativ; man fängt sie am **Einsprungpunkt** ab und misst
ab dort. CP/M-Programme starten bei `0x0100` (TPA). Typischer Ablauf:

```text
# 1) bis das OS bereit ist, dann den Programmstart abfangen
b 0x0100              # CP/M-TPA-Einsprung (sobald format.com geladen ist)
g                     # bootet, lädt das OS; tippe ggf. den Aufruf:
keys format\r         #   startet FORMAT — läuft bis Breakpoint 0x0100
mark                  # Null-Marke = Programmstart  → alle Zeiten relativ
sym format.sym        # symbolische Namen für FORMAT (optional, s. §6)

# 2) sezieren
u                     # Disassembly ab Einsprung
disp A                # A bei jedem Stopp zeigen
disp [D1BE]w          # interessante OS-Variable mitführen
n                     # Schritt für Schritt ÜBER die BDOS-/BIOS-CALLs
b BDOS if C==0x1A     # halte nur bei BDOS-Funktion 1Ah (Set DMA), bedingt
g
bt                    # wer hat gerufen?
```

**ZVE2/DMA mitverfolgen.** Sobald das Programm Disketten-I/O auslöst, übernimmt
ZVE2. Setze parallel ZVE2-Breakpoints und beobachte den Floppy-Port:

```text
b2 0x1F7D             # 3rd-stage-Lese-Routine der ZVE2 (Beispiel)
iow 0x16             # Floppy-Datenport (jeder Lesezugriff)
g                     # läuft bis ZVE2 den Breakpoint trifft
r 2                  # beide CPUs: was tut ZVE1, was ZVE2?
s2 5                # 5 ZVE2-Instruktionen steppen
```

**Daten-Hang lokalisieren** (Muster aus der CP/A-Analyse):

```text
mark 0x37A0          # Null-Marke = OS-Eintritt
b 0xC7A3 if [D1BE]w==0   # Divide-Schleife nur bei spt=0
g ; r ; vars
```

## 6. Symbole

Eine Symboldatei macht Disassembly, `bl` und Breakpoints lesbar und erlaubt
Namen als Adress-Argumente. Format: eine Zeile pro Symbol, `#` = Kommentar.
Akzeptiert werden **beide** Spaltenreihenfolgen sowie `=`:

```
# adresse name            ODER   name adresse           ODER  name = adresse
0437  STAGE1
DRIVE_INIT  0194
SELDSK = D200
```

Laden: `-s datei.sym` beim Start, `sym datei.sym` zur Laufzeit oder einzeln
`sym add NAME 0x1234`. Danach:

```text
b STAGE1            # Breakpoint per Name
gu DRIVE_INIT      # Symbol als Adresse
u STAGE1 20        # Disassembly zeigt <STAGE1> und annotiert Sprungziele <…>
```

Sprung-/Call-Ziele im Disassembly werden automatisch mit `<name>` annotiert, wenn
ein Symbol für die Zieladresse existiert.

## 6a. Listings (`.prn`) — kommentierter Original-Quelltext im Disasm

Die CP/A-Systemquellen (BIOS, BDOS, …) liegen als gut kommentierte **MACRO-80-
`.prn`-Listings** vor (z. B. `CPA_Workbench/build/bios.prn`). Lädt man das passende
Listing, hängt der Debugger an **jede** Disassembler-Zeile, deren PC im Listing steht,
die **Original-Quellzeile** (Label, Mnemonic, `;Kommentar`) an — kein Raten mehr, was
ein Codestück tut:

```text
(dbg) u 0xD227 2
  D227: C3 F3DD   JP $DDF3   ; BIOS27: JP read  ;oA Resultat; A=0 ok, A=1 sonst
  D22A: C3 82DD   JP $DD82   ; BIOS2A: JP write ;oA Resultat; A=0 ok, A=1 sonst
```

Die Annotation erscheint in `u`, an jedem Stopp/Step (`=>`) und in der Trace-Zeile
(dort ans **Zeilenende**, damit die Registerspalten ausgerichtet bleiben).

Laden: `-l bios.prn` beim Start oder `lst bios.prn` zur Laufzeit (beides wiederholbar —
mehrere Listings decken zusammen unterschiedliche Adressbereiche ab). `lst list` gibt
die geladene `Adresse → Quelle`-Tabelle aus.

**Labels als Symbole.** Beim Laden werden alle `.prn`-Labels (`name:`) automatisch als
Symbole registriert (ohne bestehende `-s`-/User-Symbole zu überschreiben). Damit sind sie
überall als Adresse nutzbar: `b BIOS27`, `gu seldsk`, `list read`, `u conin`, und `bl`/das
Disassembly zeigen die Namen. Quelltext um ein Label gibt's mit `list <label>` (s. §4).

**Adress-Offset.** Das Listing trägt absolute Lade-Adressen. Läuft der Code reloziert an
einer anderen Adresse, gibt man einen vorzeichenbehafteten Offset an, der zu jeder
Listing-Adresse addiert wird (Listing-Adresse + Offset = Laufzeit-Adresse):

```text
k1520dbg -l bios.prn@0x100 …      # Start: Listing-Adresse D227 → matcht Laufzeit-PC D327
(dbg) lst stage3.prn@-0x800       # zur Laufzeit, @-Syntax …
(dbg) lst stage3.prn -0x800       # … oder Offset als eigenes Argument
```

`OFFSET` versteht `0x1F00`, `1800h`, `512`, `-0x100`. Jedes Listing hat seinen eigenen
Offset (er wird beim Laden auf die Tabelle angewandt), mehrere relozierte Stages sind
also gleichzeitig ladbar.

**Reichweite & Grenzen.** Das Listing trägt **absolute** Lade-Adressen; ein BIOS-Listing
deckt also nur den BIOS-Adressbereich (≈ `0xD200+`) ab, nicht Boot-ROM, Bootloader-Stages
oder CCP/BDOS — dafür lädt man deren eigene Listings dazu. Wo keine `.prn`-Zeile passt,
bleibt es bei der eingebauten Disassembler-Ausgabe. Vor dem Boot (BIOS noch nicht im RAM,
Lesungen = `0xFF` → `RST 38H`) zeigt die Annotation bereits, was dort *gehört*.

> Parser: `tools/prn_listing.h` (header-only, Unit-Tests in
> `tests/cpp/test_prn_listing.cpp`). Eine Listing-Zeile zählt als Code, wenn auf die
> 4-stellige Hex-Adresse mindestens ein emittiertes Objektbyte folgt; `equ`/`aset`-
> Zeilen (führende 4-Hex = *Wert*, keine Bytes) werden so automatisch ausgesiebt.

## 7. Der relative Zyklen-Marker (`mark`)

`mark` nullt den Zyklenzähler — der Kern der Post-Boot-Analyse. Zwei Formen:

* `mark` — **jetzt** sofort nullen (am aktuellen Stopp).
* `mark <A>` — **scharfschalten**: der Nullpunkt wird gesetzt, sobald ZVE1 das
  erste Mal die Adresse `A` ausführt (z. B. OS-Eintritt `0x37A0`, Programmstart
  `0x0100`). Praktisch in Skripten, um ohne vorheriges Anhalten ab einem
  definierten Ereignis zu messen.

Danach erscheinen alle Zeiten als `+<relzyklen>`, und die Statuszeile zeigt `(rel)`.

## 8. Tipps, Grenzen, Fallstricke

* **Low-RAM 0x0000–0x07FF** ist die von ZVE1/ZVE2 geteilte Boot-/DMA-Region. Pokes
  dorthin können von der laufenden DMA überschrieben werden — zum Experimentieren
  besser freien RAM (z. B. `0x6000`) verwenden.
* **`s2` braucht eine DMA-Phase.** Läuft `/BUSRQ` nicht (ZVE2 `reset`/`wait`), tut
  `s2` nichts und meldet das. ZVE2 sinnvoll über `b2 <addr>` am DMA-Einsprung fangen.
* **`bt` ist jetzt exakt** (CALL/RST/RET-Mitschnitt, §9); `bt scan` greift weiter auf
  den alten heuristischen Stack-Scan zurück (für Stellen ohne Aufruf-Historie, z. B.
  unmittelbar nach `restore`).
* **`fin` braucht einen sauberen Rahmen.** Es hält an, sobald `SP` über den Stand
  beim Aufruf steigt; bei kaputtem Stack (z. B. nach `RST 38H` ins Leere) läuft es
  in den Sicherheits-Cap (~400 M Zyklen).
* **Bedingte Breakpoints** werten den Ausdruck bei *jedem* PC-Treffer aus — günstig,
  aber bei einem Treffer in einer engen Schleife summiert sich das. Trefferzähler in `bl`.
* **`g` ohne Argument** hat einen Sicherheits-Cap von ~400 M Zyklen, damit eine
  Sitzung nicht ewig hängt; `g <N>` läuft exakt `N` Zyklen.
* Für **vollständige Listings** mit Labels/Kommentaren bleibt `tools/z80_disasm2.py`
  kanonisch (siehe `tools/z80_disasm.md`); der eingebaute Disassembler ist für die
  interaktive Arbeit gedacht.

## 9. Reverse-Debugging, Snapshots & History-Backtrace

Drei zusammengehörige Funktionen, die den „ich bin einen Schritt zu weit / wie kam ich
hierher"-Schmerz adressieren — und die der Emulator ungewöhnlich billig macht.

### Reverse-Step (`rs`) und der Snapshot-Ring

Vor **jedem** Vorwärts-Kommando (`g`/`c`, `gu`, `s`, `s2`, `n`, `fin`, `keys`) legt der
Debugger automatisch einen Voll-Snapshot in einen Ring (Tiefe 200). `rs [N]` macht die
letzten `N` dieser Kommandos rückgängig:

```text
(dbg) g 5000000      # läuft bis irgendwohin
(dbg) s 3            # 3 Schritte zu weit gestept
(dbg) rs             # → exakt zurück auf den Zustand vor dem 's 3'
(dbg) rs 2           # zwei Kommandos zurück
```

### Benannte Snapshots (`snap` / `restore`)

```text
(dbg) snap vorher     # aktuellen Zustand sichern
(dbg) g 200000        # etwas ausprobieren …
(dbg) restore vorher  # … und exakt zurückspringen (kein Reboot)
(dbg) snap list
```

### History-Backtrace (`bt`)

`bt` nutzt einen **mitlaufenden** Aufrufstapel (jede ZVE1-Instruktion klassifiziert die
vorige: CALL/RST → push, RET/RETI → pop, „taken" am tatsächlichen PC erkannt). Das ist
**exakt** für gewöhnliche CALL/RET-Verschachtelung — kein Stack-Raten mehr:

```text
(dbg) bt
  #0 1E62
  #1 1E3D (call → 1E44, ret 1E40)  <label>  ; <.prn-Quelle>
  #2 1DCB (call → 1E2A, ret 1DCE)
```

### Modell & Grenzen (wichtig)

* Ein Snapshot = **64 KB RAM + beide Z80-Registersätze + Run-Loop-Flags**. Er enthält
  **nicht** die Geräte-Interna (CTC/PIO/SIO-Zähler, K5122-Kopfposition) und **nicht**
  das Boot-ROM-Mapping. Capture/Restore ist damit **exakt für CPU + RAM** — der typische
  Fall beim interaktiven Stepping in geladenem Code. `restore` *mitten in* einer aktiven
  DMA-/Timer-Phase kann beim Weiterlaufen driften; `restore` über die Boot-ROM-Abschalt-
  Grenze hinweg meldet eine Notiz (RAM+Register werden trotzdem gesetzt).
* Nach `rs`/`restore` wird der **Aufruf-Stack-Mitschnitt geleert** (er lässt sich nicht
  rekonstruieren) — `bt` zeigt dann erst wieder Frames, sobald du weiterläufst/steppst;
  `bt scan` funktioniert sofort.
* **Interrupt-Frames** (IM2-Vektor/NMI) erscheinen (noch) nicht im History-`bt`; sie
  korrumpieren den Stack aber nicht. Gewöhnliche CALL/RET-Nester sind exakt.
* `rs` erfasst nur ZVE1-getriebene Kommandos im Ring; der Ring hat Tiefe 200 (älteste
  fallen heraus).

> Parser/Implementierung: `tools/callstack_tracker.h` (Tracker, header-only, Tests
> `tests/cpp/test_callstack_tracker.cpp`) und `A5120Machine::captureState/restoreState`
> (Snapshot, Tests `tests/cpp/test_machine_snapshot.cpp`).
