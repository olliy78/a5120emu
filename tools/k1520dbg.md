# k1520dbg — interaktiver Debugger für die A5120/K1520-Emulation

`k1520dbg` ist ein gdb-artiger Kommandozeilen-Debugger um `A5120Machine` (den
K1520-Core). Er debuggt **beide Z80** der ZRE/K2526 gleichzeitig — die Haupt-CPU
**ZVE1** *und* den DMA-Prozessor **ZVE2** — sowie jedes geladene Programm
(Bootloader, CP/A, `format.com`, …), das nativ auf der Maschine läuft.

> **k1520dbg vs. boot_trace.** `boot_trace` ist ein nicht-interaktiver Boot-Tracer
> (absolute Zyklen ab Power-On, Report am Ende) — gut, um die *grobe Phase* eines
> Problems zu finden. `k1520dbg` arbeitet mit **Breakpoints, Single-Step, Snapshots
> und einer interaktiven Sitzung** — zum *Sezieren*. Praxis-Rezepte für beide:
> **`tools/how_to_debug_and_trace.md`**.

---

## 1. Bauen, Aufruf, Exit-Code

```sh
cmake --build build --target k1520dbg -j

./build/k1520dbg disks/cpadisk01.img                       # interaktiv
printf 'b 0xC7A3\ng\nr\nq\n' | ./build/k1520dbg disk.img   # Pipe (Skript/Agent)
./build/k1520dbg disk.hfe -x skript.dbg                     # Skriptdatei
```

```
k1520dbg [DISK] [-x SCRIPT]… [-s SYMFILE]… [-l LISTING.prn[@OFFSET]]…
  DISK        .img/.hfe → auf A: gemountet (Format cpa780/cpa800 autodetektiert)
  -x SCRIPT   Kommandos aus Datei, dann stdin   (wiederholbar)
  -s SYMFILE  Symboltabelle laden (§6)          (wiederholbar)
  -l LISTING  MACRO-80-.prn-Listing → Disasm mit Originalquelle annotieren (§6, wiederholbar)
```

Kommandos kommen aus den `-x`-Skripten (zuerst) und danach von stdin; `#`-Zeilen sind
Kommentare, `q`/EOF beendet. **Exit-Code:** `1`, wenn ein angefordertes Disk-Image nicht
gemountet werden konnte (Sitzung läuft trotzdem), sonst `0`.

**Optional GNU readline.** Wird `libreadline` zur Build-Zeit gefunden (CMake meldet es;
`apt install libreadline-dev`), bietet die interaktive Sitzung am TTY Zeileneditierung,
Kommando-History (Pfeiltasten) und Tab-Vervollständigung. Sonst nahtloser `getline`-
Fallback; Pipe-/Skriptbetrieb ist nie betroffen.

## 2. Grundkonzepte

* **Zwei CPUs.** ZVE1 ist die Haupt-CPU; ZVE2 der DMA-Prozessor, der nur läuft, während
  `/BUSRQ` gehalten wird (Floppy-/Bootsektor-Transfers). Befehle ohne Suffix betreffen
  ZVE1; ZVE2-Varianten enden auf `2` (`b2`, `s2`, `set 2`). `r 2` und die Statuszeile
  zeigen den ZVE2-Zustand (`reset`/`wait`/`run`, `BUSRQ`).
* **Break-before-execute.** Der Trace-Callback feuert **vor** der Instruktion. An einem
  Stopp steht der PC also *auf* dem noch nicht ausgeführten Befehl. `=>` markiert die
  nächste Instruktion.
* **Zeitbasis.** `cyc` sind ZVE1-Taktzyklen (instruktionsgenau, monoton ab Power-On).
  `mark` (§5) setzt einen relativen Nullpunkt; danach erscheinen Zeiten als `+<relzyklen>`.
* **Eingebauter Disassembler** (`tools/z80dis_min.h`, volle Z80-Tabelle inkl. ED/CB/DD/FD).
  Jeder Stopp/Step, `u` und `x/i` disassemblieren live; Sprungziele werden mit `<symbol>`
  annotiert, geladene `.prn`-Quelle als `; …` ans Zeilenende (§6).

## 3. Zahlen- und Ausdruckssyntax

| Kontext | Default | Beispiele |
|---------|---------|-----------|
| **Adressen/Zähler** (`b`, `gu`, `d`, `u`, `g N`, `x`, …) | dezimal (base-0) | `0xC7A3`, `0C7A3H`, `STAGE1`, `SELDSK+5` |
| **Poke-Bytes** (`e`) | **hex** | `e 0x6000 CD 10 60` |

Hex immer mit `0x…`/`…H` schreiben (`d 6000` = dezimal 6000 = 0x1770!). Symbol-/`.prn`-
Labelnamen sind überall als Adresse erlaubt.

**Ausdrücke** (für `b … if`, `disp`, `logpoint`, `x`-Adresse) — vollwertiger Evaluator:

```
REG        A F B C D E H L  AF BC DE HL IX IY SP PC  I R     (groß/klein egal)
(rr)       (HL) (DE) (BC) (SP)         Byte an der Pair-Adresse
[expr]     Byte im Speicher           Index ist selbst ein Ausdruck, z.B. [HL+5]
[expr]w    16-Bit-Wort (LE)
zahl/sym   Konstante (0x.., ..H, dez) / Symbol / .prn-Label
Operatoren + - * / %   & | ^   << >>   unär - ~   Vergleiche == != < > <= >=
           Präzedenz wie in C; Klammern ( )
```

**Bedingung** (`b … if <expr>`): feuert, wenn der Ausdruck **≠ 0** ist (Vergleiche → 0/1),
z. B. `A==0xA9`, `[03F8]!=3`, `(HL & 0xFF)==0xF7`, `[D1BE]w < 0x100`.
Leerzeichen sind in `if`/`disp` erlaubt; bei `x`/`logpoint` besser ohne (`HL+5`, `[0x6000]w`).

## 4. Kommandoreferenz

### Ablaufsteuerung
| Befehl | Wirkung |
|--------|---------|
| `g`/`c` `[N]` | weiterlaufen bis Breakpoint (oder genau `N` ZVE1-Zyklen). `g` ohne `N`: Cap ~400 M Zyklen |
| `gu <A>` | bis ZVE1 die Adresse `A` erreicht (einmaliger Temp-BP) |
| `s [N]` | `N` ZVE1-Instruktionen **hinein**-steppen (Default 1) |
| `s2 [N]` | `N` **ZVE2**-Instruktionen steppen (nur wirksam während `/BUSRQ`/DMA) |
| `n [N]` | `N` ZVE1-Instruktionen **über**-steppen (`CALL`/`RST`/`LDIR`,`INIR`… bis zum Folgebefehl) |
| `fin` | aus der aktuellen Funktion **heraus**-laufen (bis `SP` über den Rahmen steigt) |

### Reverse-Debugging & Snapshots (§7)
| Befehl | Wirkung |
|--------|---------|
| `rs [N]` | **Reverse-Step**: die letzten `N` Vorwärts-Kommandos rückgängig machen (auto-Snapshot-Ring, Tiefe 200) |
| `snap <name>` / `restore <name>` / `snap list` | benannten Voll-Snapshot speichern / wiederherstellen / listen |

### Breakpoints
| Befehl | Wirkung |
|--------|---------|
| `b <A> [if <cond>]` / `b2 <A> [if <cond>]` | Breakpoint auf ZVE1 / ZVE2 (optional bedingt, §3) |
| `tb <A>` | einmaliger (selbstlöschender) Breakpoint (ZVE1) |
| `bd <A>` / `bd2 <A>` | Breakpoint löschen (ZVE1 / ZVE2) |
| `be <A>` / `bdis <A>` (`be2`/`bdis2`) | Breakpoint **aktivieren** / **deaktivieren** (behalten, aber inaktiv) |
| `bi <A> <N>` (`bi2`) | die nächsten `N` Treffer **ignorieren** (gdb `ignore`) |
| `bint` / `bnmi` / `breti` `[on\|off]` | **Ereignis-Breakpoints** (ZVE1): bei Annahme eines **Interrupts** / **NMI** (z. B. Q240-Schutzverletzung) / vor einem **RETI/RETN** — ohne Argument toggelt |
| `bl` | alle BPs + aktive Ereignis-BPs listen (Trefferzähler, `[disabled]`, `ignore=N`, Bedingung, Symbol) |

> Ereignis-BPs werden per Zustands-Signatur erkannt (kein Core-Eingriff): Interrupt =
> `IFF1` 1→0 + `SP`-Push; NMI = Sprung `0x0066` + `SP`-Push; RETI/RETN = Opcode `ED 4D`/`45`.
> `bint`/`bnmi` halten an der ersten ISR-Instruktion, `breti` vor dem RETI.

### Watchpoints (Speicher & I/O)
| Befehl | Wirkung |
|--------|---------|
| `wp` / `wpr` / `wb` `<A\|A..B> [cond]` | **drucken** bei Schreiben / Lesen, **anhalten** bei Schreiben |
| `wd <A>` / `wd all` / `wl` | Watch löschen (alle, die `A` abdecken / alle) / listen (mit Trefferzähler) |
| `iow <P>` / `iob <P>` / `iod <P>` / `iol` | I/O-Port-Watch: drucken / anhalten / löschen / listen |

**Bereich** `A..B` überwacht die ganze Region. **Wert-Bedingung** `[cond]`: `== NN` / `!= NN`
(NN = Byte, **hex**) oder `changed` (nur bei echter Wertänderung). Bsp.: `wb 0x03F8 == 3`,
`wp 0x6000..0x60FF changed`. Jeder Treffer meldet die **auslösende CPU** (`ZVE1`/`ZVE2.PC`).

### Logpoints & Trace-to-File (laufen lassen **und** mitschreiben)
| Befehl | Wirkung |
|--------|---------|
| `logpoint <A> [expr…]` (`lp`) / `lpd <A>` / `lpl` | am PC `A` **drucken und weiterlaufen** (gdb `dprintf`): PC + Disasm + Ausdrücke / löschen / listen |
| `trace <file> [lo hi]` / `trace off` | **jede** ausgeführte Instruktion (ZVE1+ZVE2) disasm+`.prn` in `file`; optional nur im PC-Fenster `[lo,hi]`; Cap 2 Mio. Zeilen |

### Inspektion
| Befehl | Wirkung |
|--------|---------|
| `r [2]` | ZVE1-Register (+Flags `SZHPNC`, Schattenregister, `->`Rücksprung); `r 2` auch ZVE2 |
| `rj` | ZVE1-Register als **eine JSON-Zeile** (`pc,sp,af,…,cyc,rom,busrq,zve2`) — maschinenlesbar |
| `bt [N]` | **exakter** Backtrace aus dem CALL/RST/RET-Aufrufstapel (§7); `bt scan` = alter Heuristik-Scan |
| `d <A> [N]` | Hex+ASCII-Dump (Default 64 Byte) |
| `u [A] [N]` | `N` Instruktionen ab `A` disassemblieren (Default 12 ab PC; ohne `A` Fortsetzung) |
| `x/<N><fmt><sz> <A>` | gdb-artiges **Examine** (s. u.); ohne `<A>` Fortsetzung |
| `list`/`l [A] [N]` | **`.prn`-Quelltext** um `A` (Default PC/weiter), deckende Zeile mit `=>` markiert (§6) |
| `e <A> <b…>` | Bytes **poken** (hex), z. B. `e 0x6000 3E 00 C9` |
| `set [2] <reg> <v>` | Register setzen: `set HL 0x6000`, `set 2 A 5` |
| `disp <expr>` / `disp` / `undisp <n>` | Ausdruck an jedem Stopp anzeigen / Liste / Eintrag entfernen |
| `vars` | benannte CP/A-RAM-Variablen (`[03F8]`-Done-Flag, DPB-Felder) |
| `dev` | **K5122**-Floppy: Laufwerk, Zylinder/Kopf, `READING`, `headPos`/Spurlänge, Sektorgröße, `/BUSRQ` |
| `dev ctc` / `dev pio` / `dev sio` / `dev sio2` | **Chip-Zustand** + Daisy-Chain: CTC (Kanäle, IUS/IEI), BS-PIO (Ports), SIO Tastatur/DFÜ |

**`x` — Speicher in beliebigem Format** (`x/<count><fmt><size> <addr>`):
`fmt` = `x` hex · `d` signed-dez · `u` unsigned-dez · `c` Zeichen · `t` binär · `o` oktal ·
`a` Adresse(+Symbol) · `i` Instruktion(disasm+`.prn`) · `s` NUL-String. `size` = `b` Byte ·
`w`/`h` Wort (2 Byte, LE). `fmt`/`size`/`count` **bleiben erhalten** (gdb-Stil). Adresse kann
ein Ausdruck sein (`x/4xw HL`). Bsp.: `x/8xb 0xD1B0` · `x/4dw HL` · `x/10i 0x1E40` · `x/s 0xF800`.

### Speicher & Zustand laden/sichern
| Befehl | Wirkung |
|--------|---------|
| `load <f> <A>` / `save <f> <A> <N>` | Binärdatei in den RAM / RAM-Bereich in Datei |
| `savestate <f>` / `loadstate <f>` | **Maschinenzustand** (64 KB RAM + beide Z80 + ROM-Mapping + Tastatur-Subsystem → Tastatur geht nach Load) sichern/laden — einmal booten, dann ab Checkpoint fortsetzen (§7) |

### Symbole, Listings & Sonstiges
| Befehl | Wirkung |
|--------|---------|
| `sym <f>` / `sym add <name> <A>` / `sym list` | Symboldatei laden / einzeln setzen / listen (§6) |
| `lst <f.prn>[@off]` / `lst list` | `.prn`-Listing laden (annotiert Disasm, importiert Labels) / Tabelle ausgeben (§6) |
| `mark [A]` | relativen Zyklenzähler **jetzt** nullen, oder bei Erreichen von `A` scharfschalten (§5) |
| `keys <text>` | Text tippen, Maschine läuft dabei weiter (`\r`=Enter, `\t`=Tab, `\e`=ESC) |
| `screen` | 80×24-Text-VRAM (ab `0xF800`) ausgeben |
| `reset` | Power-Cycle (Boot-ROM wieder aktiv) |
| `alias <name> <expansion…>` / `unalias` / `alias` | Kommando-Kürzel definieren / löschen / listen |
| `source <f>` | Kommandos aus Datei **jetzt** abarbeiten (gdb `source`) |
| `q`/`quit` | beenden |

> **Sitzungs-Log**: alle Ausgaben gehen auf `stderr` → `… 2>&1 | tee sitzung.log`.

## 5. Der relative Zyklen-Marker (`mark`)

* `mark` — **jetzt** nullen (am aktuellen Stopp).
* `mark <A>` — **scharfschalten**: Nullpunkt wird gesetzt, sobald ZVE1 erstmals `A`
  ausführt (z. B. OS-Eintritt, Programmstart `0x0100`). Praktisch in Skripten, um ohne
  Anhalten ab einem definierten Ereignis zu messen.

Danach erscheinen Zeiten als `+<relzyklen>`, die Statuszeile zeigt `(rel)`.

## 6. Symbole & `.prn`-Listings

**Symboldatei** (`-s` / `sym`): eine Zeile pro Symbol, `#`=Kommentar; beide Spaltenreihen-
folgen sowie `=` werden akzeptiert (`0437 STAGE1` · `DRIVE_INIT 0194` · `SELDSK = D200`).
Namen sind danach überall als Adresse nutzbar; Sprung-/Call-Ziele werden im Disasm mit
`<name>` annotiert.

**`.prn`-Listings** (`-l` / `lst`). Die CP/A-Systemquellen (BIOS, BDOS, …) liegen als
kommentierte MACRO-80-`.prn`-Listings vor (z. B. `CPA_Workbench/build/bios.prn`). Geladen,
hängt der Debugger an jede Disasm-/Trace-Zeile, deren PC im Listing steht, die **Original-
Quellzeile** (Label, Mnemonic, `;Kommentar`) an:

```text
(dbg) u 0xD227 1
  D227: C3 F3DD   JP $DDF3   ; BIOS27: JP read  ;oA Resultat; A=0 ok, A=1 sonst
```

* **Labels werden Symbole.** Beim Laden werden alle `.prn`-Labels (`name:`) als Symbole
  registriert (ohne `-s`-/User-Symbole zu überschreiben) → `b BIOS27`, `list read`,
  `u conin` funktionieren direkt.
* **Adress-Offset `@OFFSET`** (signiert, `0x..`/`..h`/dez). Listing-Adresse + Offset =
  Laufzeit-Adresse, für reloziert geladenen Code: `-l bios.prn@0x100`, `lst stage3.prn@-0x800`
  bzw. `lst stage3.prn -0x800`. Jedes Listing hat seinen eigenen Offset; mehrere gleichzeitig.
* **Grenzen.** Absolute Adressen → ein BIOS-Listing deckt nur den BIOS-Bereich (~`0xD200+`,
  plus tief gemappte BIOS-Teile) ab; Boot-ROM/Stages/CCP/BDOS brauchen eigene Listings.
  Wo keine `.prn`-Zeile passt, bleibt es bei der reinen Disasm-Ausgabe.

## 7. Reverse-Debugging, Snapshots, Save-State & History-`bt`

Diese Funktionen adressieren „einen Schritt zu weit / wie kam ich hierher / nicht schon
wieder booten" — beim Emulator ungewöhnlich billig.

* **Reverse-Step.** Vor *jedem* Vorwärts-Kommando (`g`/`c`, `gu`, `s`, `s2`, `n`, `fin`,
  `keys`, `restore`) legt der Debugger einen Voll-Snapshot in einen Ring (Tiefe 200);
  `rs [N]` macht die letzten `N` Kommandos rückgängig.
* **Benannte Snapshots.** `snap <name>` / `restore <name>` — ausprobieren und exakt
  zurückspringen, ohne Reboot.
* **Save-State auf Platte.** `savestate <f>` / `loadstate <f>` (k1520dbg), `--save-state` /
  `--load-state` (boot_trace) persistieren denselben Zustand in eine Datei: **einmal booten,
  Zustand sichern, danach beliebig oft ab dort fortsetzen** statt neu zu booten.
* **History-`bt`.** Ein mitlaufender CALL/RST/RET-Aufrufstapel liefert einen **exakten**
  Backtrace (`#1 1E3D (call → 1E44, ret 1E40)`); `bt scan` greift auf den alten Heuristik-
  Scan zurück (für Stellen ohne Aufrufhistorie, z. B. direkt nach `restore`).

**Modell & Grenzen (wichtig).** Ein Snapshot = **64 KB RAM + beide Z80-Registersätze +
ROM-Mapping + Tastatur-Subsystem** (System-CTC, BS-PIO, Baud-CTC, Tastatur-SIO, K7637) —
exakt für **CPU + RAM**, und die **Tastatur funktioniert nach `loadstate`** (`keys …` am
geladenen CCP-Prompt werden ausgeführt). **Nicht** enthalten: der **Floppy-K5122** (Kopf-
position/Format) und das **K7024-VRAM** — disk-zugreifende Befehle bzw. der Bildschirm-
inhalt können nach `loadstate` abweichen; `restore`/`loadstate` *mitten in* aktiver DMA-/
Timer-Phase kann driften. Nach `rs`/`restore`
wird der Aufruf-Stack-Mitschnitt geleert (`bt` baut ihn beim Weiterlaufen neu auf; `bt scan`
geht sofort). Interrupt-Frames erscheinen (noch) nicht im History-`bt` (korrumpieren ihn
aber nicht).

## 8. Grenzen & Fallstricke

* **Low-RAM 0x0000–0x07FF** ist die von ZVE1/ZVE2 geteilte Boot-/DMA-Region — Pokes
  dorthin können von laufender DMA überschrieben werden; zum Experimentieren freien RAM
  (z. B. `0x6000`) nutzen.
* **`s2` braucht eine DMA-Phase** (`/BUSRQ` aktiv); sonst tut es nichts und meldet das.
  ZVE2 sinnvoll mit `b2 <addr>` am DMA-Einsprung fangen.
* **`fin` braucht einen sauberen Rahmen**; bei kaputtem Stack läuft es in den
  Sicherheits-Cap (~400 M Zyklen).
* **Bedingte Breakpoints** werten den Ausdruck bei *jedem* PC-Treffer aus (Trefferzähler in `bl`).
* Für **vollständige statische Listings** bleibt `tools/z80_disasm2.py` kanonisch
  (`tools/z80_disasm.md`); der eingebaute Disassembler ist für die interaktive Arbeit.

## 9. Spickzettel

```
g/c [N]  gu A  s [N]  s2 [N]  n [N]  fin            ← laufen/steppen
rs [N]   snap name | restore name | snap list       ← reverse / snapshots
savestate f | loadstate f                            ← zustand auf platte (kein reboot)
b A [if cond]  b2 A  tb A  bd/bd2 A  be/bdis A  bi A N  bl   ← breakpoints (…2 = ZVE2)
bint | bnmi | breti [on|off]                         ← break on interrupt/NMI/RETI
wp/wpr/wb A|A..B [==v|!=v|changed]  wd A|all  wl  iow/iob P  iod P  iol  ← watch
logpoint A [expr..]  lpd A  lpl   trace f [lo hi]|off ← log ohne anhalten
r [2]  rj  bt [N]|bt scan  d A [N]  u [A][N]  x/Nfs A  list/l [A][N]  e A b..  vars
x fmt: x d u c t o a i s   size: b w   (z.B. x/8xb A · x/4dw HL · x/10i A · x/s A)
dev [ctc|pio|sio|sio2]   set [2] reg v   disp expr  undisp n   load/save f A [N]
mark [A]   sym f|add|list   lst f.prn[@off]   keys txt  screen  reset
alias name expansion | unalias name | alias   source f   q
expr: REG (rr) [expr] [expr]w  + - * / % & | ^ << >> ~  ( )  == != < > <= >=
.prn-Labels werden Symbole → b/gu/u/list <label> direkt (z.B. b BIOS27)
```
