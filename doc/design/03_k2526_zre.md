# Feinentwurf: K2526 ZRE – Zentrale Recheneinheit

**Modul:** `core/cards/k2526/`  
**Dateien:** `k2526.h`, `k2526.cpp`, `rom_data.h` (generiert)  
**Dokumentation:** `doc/trascripted/Zentrale Recheneinheit_K2526_K2527.md`

## 1. Aufgabe

Die K2526 ist die Hauptprozessorkarte des A5120. Sie enthält:
- **ZVE1**: Haupt-Z80-CPU (U880D, 2.4576 MHz)
- **ZVE2**: Zweite Z80-CPU als DMA-Prozessor
- **Lade-ROM**: 1 KB EPROM bei 0000H–03FFH
- **CTC** (Q 302): 4 Kanäle, Timer/Zähler, I/O 0CH–0FH
- **BS-PIO** (Q 301): Steuerfunktionen, I/O 08H–0BH
- **U-Bus**: 4 spezielle I/O-Adressen (00H–07H)
- **Speicher-/E/A-Schutz** (Q 240): 1024-Segment-Schutztabelle
- **Taktgenerator**: 9832 kHz Quarz → 2.4576 MHz

**Emulationsumfang:** Die K2526-Klasse emuliert ZVE1 (Haupt-CPU), ZVE2 (DMA-CPU),
Lade-ROM, BS-PIO, CTC, U-Bus-Register und den Q240-Speicher-/E/A-Schutz vollständig.
ZVE1 (`Z80 cpu_`) und ZVE2 (`Z80 zve2_`) sind private Member der Klasse K2526.
ZVE2 teilt den K1520-Bus mit ZVE1 und führt Sektor-Transfers durch, wenn /BUSRQ assertiert ist.

## 2. Interne Architektur

```
K2526-Karte
├── Z80CPU (ZVE1)           – Haupt-CPU (ist Bestandteil von K2526, nicht A5120Machine)
├── Z80CPU (ZVE2)           – DMA-CPU (teilt K1520-Bus; aktiv während BUSRQ assertiert)
├── Z80CTC                  – Ports 0CH-0FH
├── Z80PIO (BS-PIO)         – Ports 08H-0BH
├── EPROMDevice (1KB)       – Lade-ROM, 0000H-03FFH (abschaltbar)
├── UBus-Register           – Ports 00H-07H (Tastatur, Lampen, INT-BS, RES-SPA)
└── MemIOProtect (Q240)     – 1024-Segment-Schutztabelle (implementiert)
```

## 4. I/O-Port-Mapping (00H–0FH)

| Port | Richtung | Funktion | Emulation |
|------|----------|----------|-----------|
| 00H | W | /INT-BS: OS-Ebenen-Wechsel-Interrupt auslösen | `int_bs_active_ = true`; BS-PIO B1=0 |
| 01H | R | /UCS2: Tastengültigkeitsabfrage | `ubus_kbd_valid_` |

## 5. BS-PIO Funktionen

### 5.1 Port A (Richtungsmaske 0x7F – Bit 7 ist Ausgang)

| Bit | Signal | Beschreibung |
|-----|--------|--------------|
| 0 | /M1 | Einzelbefehlsabarbeitung (Interrupt-Vektor für ZVE1-Einzelschritt) |
| 1 | /SUE | Spannungsüberwachungseingang |
| 2 | /NMI | NMI-Quelle (Netzausfall / E/A-Schutz-Verletzung) |
| 3 | SPS | Speicherschutz-Verletzungsanzeige (1 = Verletzung erkannt) |
| 4 | /EBF | Einzelbefehlsende |
| 5 | /WR | Speichertest (WR-Signal für Speicherkarten-Test) |
| 6 | /RDY | Speicher/Peripherie bereit |
| 7 | MEMDI1/2 | Speicherbereich-Umschaltsignal **(Ausgang!)**: A7=1 → `bus_.setMEMDI(true)` |

### 5.2 Port B (Ausgänge; Richtungsmaske 0xE2 – B0/B2/B3/B4 sind Ausgänge, B1/B5–B7 Eingänge)

| Bit | Signal | Beschreibung | Emulation |
|-----|--------|--------------|-----------|
| 0 | /LD-ROM | Lade-ROM abschalten (active low: B0=0 → ROM aus) | `rom_enabled_ = (data & 1) != 0` |
| 1 | /INT-BS | OS-Ebenen-Wechsel-Anforderung **(Eingang!)** | `int_bs_active_` → portBWrite B1=0 |
| 2 | /SPS-ESR | Speicherschutz-RAM Schreibgenehmigung (active low) | `spa_.enableTableWrite(!(data & 4))` |
| 3 | /WAIT-ZVE2 | WAIT-Signal für ZVE2 (active low: B3=0 → ZVE2 anhalten) | `zve2_wait_ = !(data & 8)` |
| 4 | /SA | Netzausschaltung (active low: B4=0 → Shutdown) | `shutdown_req_ = true` |
| 5–7 | Config | Brückentest **(Eingänge)**: Konfiguration des Rechners | `cfg_.config_bridges` |

### 5.3 Lade-ROM Abschaltung und BS-PIO B Ausgangsbits

Beim Systemstart ist das 1 KB Lade-ROM bei 0000H–03FFH aktiv (B0 per Pull-Up auf 1).
Das BIOS schreibt nach der Initialisierung **Bit 0 = 0** (B0=0, /LD-ROM aktiviert) in BS-PIO
Port B, um das ROM abzuschalten. Danach liegt das RAM des K3526 vollständig bei 0000H.

Gleichzeitig werden beim BS-PIO-B-Write B2, B3 und B4 ausgewertet:

```cpp
void K2526::onBSPIOPortBOut(uint8_t data) {
    // B0: /LD-ROM (active low)
    bool rom_on = (data & 0x01) != 0;
    if (!rom_on && rom_enabled_) {
        bus_.unregisterMem(&rom_);
        rom_enabled_ = false;
    }

    // B2: /SPS-ESR – ZVE1-Schreibzugriffe in Q240-Schutztabelle umleiten
    spa_.enableTableWrite(!(data & 0x04));

    // B3: /WAIT-ZVE2 – ZVE2 bei B3=0 anhalten
    zve2_wait_ = !(data & 0x08);

    // B4: /SA – Netzausschaltung bei B4=0
    if (!(data & 0x10) && !shutdown_req_) shutdown_req_ = true;
}
```

## 6. ZVE2 (DMA-CPU) – Vollständige Emulation

Die zweite CPU (ZVE2) ist als echter Z80-Prozessor (`Z80 zve2_`) in der K2526-Klasse
implementiert. ZVE2 teilt den selben K1520-Bus mit ZVE1 (gleiches RAM, gleiche I/O-Ports).
ZVE2 **umgeht den Q240-Schutz** (privilegierter DMA-Zugriff).

### 6.1 Hardware-Modell

```
K2526-Karte
├── ZVE1 (Z80 cpu_)    – Haupt-CPU, Q240-Schutz aktiv für alle Zugriffe
├── ZVE2 (Z80 zve2_)   – DMA-CPU, Q240-Schutz bypassed (privelegiert)
│   ├── readByte  → bus_.memRead()    (direkt, kein Schutz)
│   ├── writeByte → bus_.memWrite()   (direkt, kein Schutz)
│   ├── readPort  → bus_.ioRead()     (gemeinsame I/O-Ports)
│   └── writePort → bus_.ioWrite()    (gemeinsame I/O-Ports)
└── BUSRQ/BUSAK-Protokoll (Bus-Verriegelung, hardware-genau – s. §6.1b):
    ─ Floppy (FD-Adapter/K5122) assertiert /BUSRQ erst, wenn die Daten
      rotatorisch BEREIT sind (Seek-/Drehlatenz nach /STR)
    ─ ZVE1 quittiert mit /BUSAK, schaltet seine Busse tristate → INAKTIV
    ─ Run-Loop steppt dann AUSSCHLIESSLICH ZVE2 (Verriegelung) – nie beide
    ─ zve2Step() gibt 0 zurück wenn /WAIT-ZVE2 aktiv (zve2_wait_=true)
    ─ Floppy setzt /BUSRQ=1 bei Transfer-Ende → ZVE1 wieder AKTIV
```

### 6.1b Bus-Verriegelung ZVE1↔ZVE2 (hardware-genau)

> **Quelle:** `doc/trascripted/Zentrale Recheneinheit_K2526_K2527.md` §6.7
> („Programmierbarer DMA-Kanal", Z.551–585) und §6.5 („WAIT-Einblendung für ZVE2").

Auf der **echten K2526** sind ZVE1 (A3) und ZVE2 (A8) **gegenseitig verriegelt** und
arbeiten **komplementär** – es ist **immer genau eine** CPU aktiv, umgeschaltet durch
`/BUSRQ` (transkribierte Doku §6.7.2):

| `/BUSRQ` | aktiv | inaktiv (Bus tristate) | Beleg |
|----------|-------|------------------------|-------|
| `=1` (high) | **ZVE1** | ZVE2 (Adreß-/Daten-/Steuerbus hochohmig) | Z.560, Z.571 |
| `=0` (low)  | **ZVE2** | ZVE1 (quittiert mit `/BUSAK`, Busse hochohmig) | Z.575–578 |

Ablauf eines DMA-Transfers (Z.575–579):

1. Der **Floppy (FD-Adapter)** fordert den Bus an: `/BUSRQ = 0` – **erst wenn er Daten
   bereitstellen kann** (nach Positionieren/Drehlatenz; vgl. die kontinuierliche
   Spurrotation in `07_k5122`). *Nicht* schon im Moment des `/STR`-Befehls.
2. „Die ZVE1 quittiert diese Anforderung mit … `/BUSAK-ZVE1 = 0` **nach Beenden des
   Maschinenzyklus**" → ZVE1 schaltet seine Busse tristate und ist **inaktiv**.
3. „Die 2. ZVE ist aktiv, **solange `/BUSRQ = low` bleibt**" – ZVE2 fährt den **ganzen**
   Transfer; ZVE1 steht still und kann **keinen** Bus-/I/O-Zugriff tätigen.
4. „Die Anschlußsteuerung des Floppy-Disk schaltet `/BUSRQ = 1`, wenn der **Datentransfer
   beendet** ist" → ZVE1 wird wieder aktiv und liest das Ergebnis.

`/WAIT-ZVE2` (BS-PIO B3, §6.5) ist **keine** Nebenläufigkeit, sondern blendet nur einen
`/WAIT`-Takt in die `/M1`-Zyklen der ZVE2 ein (ZVE2 darf `/M1` nicht auf den Systembus
geben – sonst käme die Interrupt-Prioritätenkette durcheinander). Es **verlangsamt** ZVE2,
schaltet aber keine zweite CPU zu.

**Konsequenz für die Emulation (100 %-Nachbildung):** Während `/BUSRQ` assertiert ist,
steppt die Run-Loop **ausschließlich ZVE2** – ZVE1 wird **nicht** nebenläufig gesteppt
(früheres 1:1-Konkurrenz-Modell war eine Näherung und ist hardware-falsch). Damit kann
ZVE1 die K5122-Steuersignale (`/STR`, Mark-Strobes) **nicht** mehr verändern, während
ZVE2 den Spur-Stream liest – das beseitigt die Bus-Konflikt-Korruption (ZVE2 las `0x00`
trotz valider Spur; `analyse_cpa_boot.md` §6b/§6c).

**Implementierungs-Stand (2026-06-17): umgesetzt.** Die Verriegelung ist als **per-Byte-
`/BUSRQ`-Modell** realisiert und ersetzt das frühere konkurrente 1:1-Stepping sowie die
sektorgrößen-abhängige `OUT(13H),03H`-Arbitrierung. Vollständige Beschreibung des Datenpfads
in **`07_k5122_afs.md` §7.1/§7.2**; hier nur die ZVE-Arbitrierungssicht:

- **K5122 bildet `/BUSRQ` pro Byte** (Doku §5.6.1, RDY des Daten-PIO): `/STR↓` → `/BUSRQ`,
  `ioRead(0x16)` gibt das Byte aus und nimmt `/BUSRQ` zurück, nach ~1 Byteperiode
  (`kBytePeriodCycles≈150`) wird das nächste Byte bereitgestellt. So **verschränken** sich
  ZVE1 und ZVE2 byteweise von selbst: während `/BUSRQ` low ist läuft ZVE2, in den Byte-Lücken
  ZVE1 — **ohne** das frühere „beide jede Iteration steppen".
- **`A5120Machine::run()` verriegelt** entsprechend: während `/BUSRQ` wird **nur** `zve2Step()`
  ausgeführt (+ `afs_.update()`), **kein** nebenläufiges ZVE1 (`continue`).
- **Transfer-Ende** = gelatchtes `/STR=1` (≥ `kStrEndSampleCycles≈320` Takte; kurze Boot-ROM-
  Setup-Strobes ≤18 Takte werden vom Datenseparator verschluckt) **plus** der programmweite
  ZVE2-Completion-Handshake `[0x03F8]=3` (Transition 0→3/Runde → `endDmaTransfer()`).

Der frühere „Drehlatenz + rotatorisches Gap-Pulsing"-Prototyp (`busrq_arm_delay_`,
`interSectorGapCycles()`) ist damit gegenstandslos — die Byte-Verschränkung liefert die
ZVE1-Lücken zwischen den Sektoren automatisch.

**Warum `[0x03F8]=3` noch nötig ist (statt reinem `/STR`):** Der **Chained Loader (round 2)**
wartet die DMA per **Zeitschleife** ab (geladener Code `0x0441 = DEC C; JR NZ`, kein
Flag-Poll). Unter der Per-Byte-Verschränkung verschiebt sich das ZVE1:ZVE2-Tempo, sodass
ZVE1s Zeitschleife **vor** ZVE2s DMA-Ende abläuft → ZVE1 zieht `/STR=1` zu früh →
unvollständiger Read → Retry-Schleife (ZVE2 dreht in IDAM-Suche `0x020B`/`0x025A`,
`/STR`-Re-Strobe `0x01FA`). Der `[0x03F8]=3`-Watcher gibt den Bus **exakt** bei ZVE2-
Completion frei und umgeht die Tempo-Abhängigkeit; er ist **kein** größenabhängiges Signal.
Ob er durch ein reines HW-Signal ersetzbar ist, hängt am echten Arbitrierungsmodus
(BUSRQ- vs. WAIT-Betrieb, s. `07_k5122_afs.md` §7.6).

**Stand:** alle Boot-Integrationstests grün, voller CP/A-Boot bis „Bitte Uhrzeit eingeben!".
Offen: Formaterkennung beim Verzeichniszugriff (`dir`) — dieselbe Completion-Frage im
laufenden OS.

### 6.2 Reset-Steuerung (Port 04H, /RES-ZVE2)

Port 04H ist ein aktiv-low Reset-Signal für ZVE2:

| Wert | /RES-ZVE2 | Verhalten |
|------|-----------|-----------|
| Bit0 = 0 | assertiert | ZVE2 in Reset, führt keine Schritte aus |
| Bit0 = 1 | freigegeben | ZVE2 startet von PC=0000H |

```cpp
case 0x04:  // /RES-ZVE2 (active-low, bit 0)
    if (!(data & 0x01) && !zve2_reset_) {
        zve2_reset_ = true;         // → ZVE2 anhalten
    } else if ((data & 0x01) && zve2_reset_) {
        zve2_reset_ = false;
        zve2_.reset();              // → ZVE2 starten (PC=0000H)
    }
```

**Beim Systemstart** ist ZVE2 im Reset. Das **Boot-ROM gibt ZVE2 aber aktiv frei**
(OUT 04H, 01H bei ROM 0x019B) und lässt es das Bootsektor-DMA-Programm (STROBE_LOOP,
ab 0x01DD) ausführen – ZVE2 ist also bereits beim Booten im Einsatz, nicht erst unter
dem Betriebssystem. Der `dmaUpdate()`-Fallback greift nur, falls ZVE2 im Reset/WAIT
bleibt. Verifizierter Ablauf: `doc/analyse_zre_rom_boot.md`.

### 6.3 DMA-Ablauf mit ZVE2 (Bootsektor lesen)

```
ZVE1 schreibt /STR mit /WE=1 (Lesen):
  → K5122::handleCtrlPortAWrite()
      doReadSector()       ← GANZE Spur als 138-Byte-Block-Stream laden (07_k5122 §7.1)
      bus_.assertBUSRQ()

A5120Machine::run() – Run-Loop:
  if (bus_.isBUSRQ()) {
      if (ZVE2 aktiv) {                      // nicht im Reset, nicht WAIT
          zre_.zve2Step();                   // ZVE2 sucht IDAMs, kopiert per INI/INIR
          if (bus_.memRead(0x03F8) == 3)     // ZVE2 hat alle Sektoren kopiert?
              afs_.endDmaTransfer();         // → BUSRQ freigeben
          // KEIN continue: ZVE1 wird ebenfalls gesteppt (Nebenläufigkeit!)
      } else {
          afs_.dmaUpdate();                  // Fallback (ZVE2 im Reset/WAIT)
          remaining--; continue;
      }
  }

ZVE2 (STROBE_LOOP, ROM 0x01DD..0x0267): durchsucht den Spur-Stream nach der IDAM
  jedes Sektors (CP cyl/head/sec/size), überträgt 128 Datenbytes per INI/INIR,
  Schleife bis sector_id==[07F2]. Abschluss: OUT(11H,03); LD ([03F8H]),3.
```

Die BUSRQ-Freigabe erfolgt **erst bei der Completion** (`[0x03F8]=3`), nicht beim
Leerlesen des Puffers. Während der DMA wird ZVE1 nebenläufig weitergesteppt, damit es
`CALL 0194` beendet ([0x03F8]=0 bei 0x01B3) und in seine Poll-Schleife 0x0168 gelangt,
bevor ZVE2 `=3` schreibt (sonst Clobber). Siehe `doc/analyse_zre_rom_boot.md`.

### 6.4 SCHREIB-DMA mit ZVE2

```
ZVE1 schreibt /STR mit /WE=0 (Schreiben):
  → K5122::handleCtrlPortAWrite()
      sector_buf_.clear()    ← Buffer leeren für ZVE2-Eingabe
      bus_.assertBUSRQ()

ZVE2-DMA-Programm:
  LD BC, sector_size
  LD HL, src_addr           ; Quelladresse im RAM
  LD C, 14H                 ; K5122 Daten-PIO Port A
  OTIR                      ; BC Bytes von (HL) → Port C, HL++, B--
  ; Abschluss-Signal: erneut /STR schreiben (während Bus gehalten)
  OUT (10H), A              ; /STR-Flanke → K5122 erkennt ZVE2-Kontext

K5122::handleCtrlPortAWrite() – ZVE2-Kontext (bus_.isBUSRQ() true):
  doWriteSector()        ← sector_buf_ in Disk-Image schreiben
  bus_.releaseBUSRQ()
```

### 6.5 Fallback: ZVE1-Byte-Polling (nur wenn ZVE2 im Reset/WAIT)

Falls ZVE2 im Reset oder durch /WAIT-ZVE2 angehalten ist, gibt es keine zweite CPU,
die die DMA fährt; dann übernimmt der `dmaUpdate()`-Fallback. ZVE1 liest den gefüllten
`sector_buf_` byteweise per `IN A,(16H)`:

```
/STR-Flanke → doReadSector() füllt sector_buf_ → BUSRQ assertiert
dmaUpdate() → BUSRQ freigeben (sector_buf_ bleibt gefüllt)
ZVE1 pollt Port 16H byte-weise → sector_buf_[buf_pos_++]
```

> **Hinweis:** Der **reale A5120-Boot nutzt diesen Fallback nicht** – das Boot-ROM
> startet ZVE2 und lässt es die Bootsektoren per DMA laden (siehe 6.2/6.3). Der
> Fallback bleibt für OS-Szenarien und Unit-Tests erhalten.

## 7. Speicher-/E/A-Schutz (Q 240) – implementiert

Die Q240-Schutzlogik unterteilt den 64 KB-Adressraum in **1024 Segmente à 64 Bytes**.
Jedes Segment hat ein Schutzbyte in der Tabelle (Bit 0 = Schreibschutz, Bit 1 = Leseschutz).

### 7.1 Klasse MemIOProtect

```cpp
class MemIOProtect {
public:
    static constexpr int SEGMENTS = 1024;

    void    enableTableWrite(bool active);  // /SPS-ESR: nächste ZVE1-Writes → Tabelle
    bool    isTableWriteEnabled() const;
    bool    isWriteProtected(uint16_t addr) const;  // Bit 0 des Segment-Eintrags
    bool    isReadProtected(uint16_t addr)  const;  // Bit 1 des Segment-Eintrags
    void    writeEntry(uint16_t addr, uint8_t flags);
    uint8_t readEntry(uint16_t addr)        const;
    void    setViolation();
    void    resetViolation();
    bool    isSPSViolation()                const;
    void    reset();

private:
    std::array<uint8_t, 1024> prot_table_{};
    bool table_write_enabled_ = false;
    bool sps_ind_             = false;
};
```

### 7.2 Schutz-Aktivierung durch /SPS-ESR (BS-PIO B2)

Wenn BS-PIO Port B Bit 2 (/SPS-ESR) auf 0 gesetzt wird, werden alle ZVE1-Schreibzugriffe
**in die Schutztabelle** umgeleitet statt ins RAM:

```
Segment-Nummer = Adresse >> 6           (Adresse / 64)
Tabellen-Eintrag = geliefertes Datenbyte
```

ZVE2 umgeht diesen Mechanismus vollständig (privilegierter DMA).

### 7.3 Schutz-Enforcement (ZVE1-Callbacks)

```cpp
// ZVE1 Lese-Callback
cpu_.readByte = [this](uint16_t a) -> uint8_t {
    if (spa_.isTableWriteEnabled()) return spa_.readEntry(a);
    if (spa_.isReadProtected(a))   { handleSPSViolation(); return 0xFF; }
    return bus_.memRead(a);
};

// ZVE1 Schreib-Callback
cpu_.writeByte = [this](uint16_t a, uint8_t d) {
    if (spa_.isTableWriteEnabled()) { spa_.writeEntry(a, d); return; }
    if (spa_.isWriteProtected(a))   { handleSPSViolation(); return; }
    bus_.memWrite(a, d);
};
```

### 7.4 Verletzungsbehandlung (handleSPSViolation)

Bei einem Schutzverstoß:
1. `sps_ind_` wird auf `true` gesetzt
2. BS-PIO Port A Bit 3 (SPS) wird auf 1 gesetzt (Input über `portAWrite`)
3. `bus_.assertNMI()` – ZVE1 erhält einen Nicht-Maskierbaren Interrupt

```cpp
void K2526::handleSPSViolation() {
    if (!sps_ind_) {
        sps_ind_ = true;
        spa_.setViolation();
        updatePortAInputs();
        bus_.assertNMI();
    }
}
```

### 7.5 Verletzung zurücksetzen (/RES-SPA, Port 02H)

Der CPU-Code quittiert einen Schutzverstoß durch Schreiben auf Port 02H:

```cpp
case 0x02:  // /RES-SPA
    spa_.resetViolation();
    sps_ind_      = false;
    int_bs_active_= false;
    updatePortAInputs();
    bus_.clearNMI();
    bs_pio_.portBWrite(buildPortBInputs());
    break;
```

### 7.6 /INT-BS (Port 00H) – OS-Ebenen-Wechsel-Interrupt

Schreiben auf Port 00H signalisiert einen gewünschten Betriebssystemebenen-Wechsel.
Die BS-PIO B1-Eingangsleitung (/INT-BS) wird dabei auf 0 gezogen:

```cpp
case 0x00:  // /INT-BS
    int_bs_active_ = true;
    bs_pio_.portBWrite(buildPortBInputs());
    break;
```

`buildPortBInputs()` setzt Bit 1 = 0, wenn `int_bs_active_` true ist:

```cpp
uint8_t K2526::buildPortBInputs() const {
    uint8_t val = (cfg_.config_bridges & 0x07) << 5;   // B5–B7: Brücken
    if (!int_bs_active_) val |= 0x02;                   // B1: /INT-BS (active-low)
    return val;
}
```

## 8. Boot-Sequenz

```
/RESET → ZVE1 startet bei 0000H, ROM aktiv (1KB bei 0000H), Q240 zurückgesetzt.
ROM-Code (verifizierte Reihenfolge, siehe doc/analyse_zre_rom_boot.md):
  1. RAM 0000–07FF löschen (LDI-Schleife), IM 2, SP, BS-PIO init
  2. RAM-Test (K3526), Disk-Tabelle kopieren
  3. ROM[00BA..03FF] → RAM kopieren (Laufzeitcode inkl. ISR-Vektoren)
  4. BS-PIO B0=0 → /LD-ROM → ROM ABSCHALTEN (0000–03FF ist nun K3526-RAM,
     gemeinsam für ZVE1+ZVE2; RAM[0000]=C3 DD 01 = JP 01DD = ZVE2-Einstieg)
  5. K5122 ctrl-PIO + Vektor (0xBA→ISR 01C7), Laufwerk wählen, Drive-Detect
  6. 4 Indexpulse abwarten, [03FD]=0x87 setzen (IDAM-Pfad)
  7. SEEK_SETUP + CALL 0194: ZVE2 freigeben (OUT 04H) und Bootsektor-DMA starten
  8. ZVE2 liest Sektoren 1–4 nach RAM[0400+], meldet [03F8]=3
  9. Signaturprüfung RAM[0400]="SYL", dann CALL 0437H → geladener Boot-Code
```

Das Lade-ROM (1 KB, `doc/EPROMS/zre.rom`) enthält diesen Bootstrap. Die zweistufige
ZVE1↔ZVE2-DMA (Schritte 7–8) ist der kritische, in 6.3 beschriebene Teil.
