# Feinentwurf: K2526 ZRE – Zentrale Recheneinheit

**Modul:** `core/cards/k2526/`  
**Dateien:** `k2526.h`, `k2526.cpp`, `rom_data.h` (generiert)  
**Dokumentation:** `doc/trascripted/Zentrale Recheneinheit_K2526_K2527.md`

---

## 1. Aufgabe

Die K2526 ist die Hauptprozessorkarte des A5120. Sie enthält:
- **ZVE1**: Haupt-Z80-CPU (U880D, 2.4576 MHz)
- **ZVE2**: Zweite Z80-CPU als DMA-Prozessor
- **Lade-ROM**: 1 KB EPROM bei 0000H–03FFH
- **CTC** (Q 302): 4 Kanäle, Timer/Zähler, I/O 0CH–0FH
- **BS-PIO** (Q 301): Steuerfunktionen, I/O 08H–0BH
- **U-Bus**: 4 spezielle I/O-Adressen (00H–07H)
- **Speicher-/E/A-Schutz** (Q 240): 1024-Segment-Schutztabelle
- **Taktgenerator**: 9832 kHz Quarz → 2.4576 MHz

**Emulationsumfang:** Die K2526-Klasse emuliert ZVE1 (Haupt-CPU), ZVE2 (DMA-CPU),
Lade-ROM, BS-PIO, CTC, U-Bus-Register und den Q240-Speicher-/E/A-Schutz vollständig.
ZVE1 (`Z80 cpu_`) und ZVE2 (`Z80 zve2_`) sind private Member der Klasse K2526.
ZVE2 teilt den K1520-Bus mit ZVE1 und führt Sektor-Transfers durch, wenn /BUSRQ assertiert ist.

## 2. Interne Architektur

```
K2526-Karte
├── Z80CPU (ZVE1)           – Haupt-CPU (ist Bestandteil von K2526, nicht A5120Machine)
├── Z80CPU (ZVE2)           – DMA-CPU (teilt K1520-Bus; aktiv während BUSRQ assertiert)
├── Z80CTC                  – Ports 0CH-0FH
├── Z80PIO (BS-PIO)         – Ports 08H-0BH
├── EPROMDevice (1KB)       – Lade-ROM, 0000H-03FFH (abschaltbar)
├── UBus-Register           – Ports 00H-07H (Tastatur, Lampen, INT-BS, RES-SPA)
└── MemIOProtect (Q240)     – 1024-Segment-Schutztabelle (implementiert)
```

## 4. I/O-Port-Mapping (00H–0FH)

| Port | Richtung | Funktion | Emulation |
|------|----------|----------|-----------|
| 00H | W | /INT-BS: OS-Ebenen-Wechsel-Interrupt auslösen | `int_bs_active_ = true`; BS-PIO B1=0 |
| 01H | R | /UCS2: Tastengültigkeitsabfrage | `ubus_kbd_valid_` |
| 02H | W | /RES-SPA: Speicherschutz-Verletzung zurücksetzen | `spa_.resetViolation()`, NMI löschen, INT-BS löschen |
| 03H | W | /UCS4: Fehlerlampe | `ubus_error_lamp_` |
| 04H | W | /RES-ZVE2: DMA-CPU Reset | `zve2_reset_` + `zve2_.reset()` |
| 05H | W | /UCS3: Lampenansteuerung | `ubus_lamp_` |
| 06H | R | /UCS1: Tastencode lesen | `ubus_kbd_code_` |
| 07H | — | reserviert | ignoriert |
| 08H | R/W | BS-PIO Port A Daten | → Z80PIO |
| 09H | R/W | BS-PIO Port A Steuer | → Z80PIO |
| 0AH | R/W | BS-PIO Port B Daten | → Z80PIO |
| 0BH | R/W | BS-PIO Port B Steuer | → Z80PIO |
| 0CH | R/W | CTC Kanal 0 | → Z80CTC |
| 0DH | R/W | CTC Kanal 1 | → Z80CTC |
| 0EH | R/W | CTC Kanal 2 | → Z80CTC |
| 0FH | R/W | CTC Kanal 3 | → Z80CTC |

## 5. BS-PIO Funktionen

### 5.1 Port A (Richtungsmaske 0x7F – Bit 7 ist Ausgang)

| Bit | Signal | Beschreibung |
|-----|--------|--------------|
| 0 | /M1 | Einzelbefehlsabarbeitung (Interrupt-Vektor für ZVE1-Einzelschritt) |
| 1 | /SUE | Spannungsüberwachungseingang |
| 2 | /NMI | NMI-Quelle (Netzausfall / E/A-Schutz-Verletzung) |
| 3 | SPS | Speicherschutz-Verletzungsanzeige (1 = Verletzung erkannt) |
| 4 | /EBF | Einzelbefehlsende |
| 5 | /WR | Speichertest (WR-Signal für Speicherkarten-Test) |
| 6 | /RDY | Speicher/Peripherie bereit |
| 7 | MEMDI1/2 | Speicherbereich-Umschaltsignal **(Ausgang!)**: A7=1 → `bus_.setMEMDI(true)` |

### 5.2 Port B (Ausgänge; Richtungsmaske 0xE2 – B0/B2/B3/B4 sind Ausgänge, B1/B5–B7 Eingänge)

| Bit | Signal | Beschreibung | Emulation |
|-----|--------|--------------|-----------|
| 0 | /LD-ROM | Lade-ROM abschalten (active low: B0=0 → ROM aus) | `rom_enabled_ = (data & 1) != 0` |
| 1 | /INT-BS | OS-Ebenen-Wechsel-Anforderung **(Eingang!)** | `int_bs_active_` → portBWrite B1=0 |
| 2 | /SPS-ESR | Speicherschutz-RAM Schreibgenehmigung (active low) | `spa_.enableTableWrite(!(data & 4))` |
| 3 | /WAIT-ZVE2 | WAIT-Signal für ZVE2 (active low: B3=0 → ZVE2 anhalten) | `zve2_wait_ = !(data & 8)` |
| 4 | /SA | Netzausschaltung (active low: B4=0 → Shutdown) | `shutdown_req_ = true` |
| 5–7 | Config | Brückentest **(Eingänge)**: Konfiguration des Rechners | `cfg_.config_bridges` |

### 5.3 Lade-ROM Abschaltung und BS-PIO B Ausgangsbits

Beim Systemstart ist das 1 KB Lade-ROM bei 0000H–03FFH aktiv (B0 per Pull-Up auf 1).
Das BIOS schreibt nach der Initialisierung **Bit 0 = 0** (B0=0, /LD-ROM aktiviert) in BS-PIO
Port B, um das ROM abzuschalten. Danach liegt das RAM des K3526 vollständig bei 0000H.

Gleichzeitig werden beim BS-PIO-B-Write B2, B3 und B4 ausgewertet:

```cpp
void K2526::onBSPIOPortBOut(uint8_t data) {
    // B0: /LD-ROM (active low)
    bool rom_on = (data & 0x01) != 0;
    if (!rom_on && rom_enabled_) {
        bus_.unregisterMem(&rom_);
        rom_enabled_ = false;
    }

    // B2: /SPS-ESR – ZVE1-Schreibzugriffe in Q240-Schutztabelle umleiten
    spa_.enableTableWrite(!(data & 0x04));

    // B3: /WAIT-ZVE2 – ZVE2 bei B3=0 anhalten
    zve2_wait_ = !(data & 0x08);

    // B4: /SA – Netzausschaltung bei B4=0
    if (!(data & 0x10) && !shutdown_req_) shutdown_req_ = true;
}
```

## 6. ZVE2 (DMA-CPU) – Vollständige Emulation

Die zweite CPU (ZVE2) ist als echter Z80-Prozessor (`Z80 zve2_`) in der K2526-Klasse
implementiert. ZVE2 teilt den selben K1520-Bus mit ZVE1 (gleiches RAM, gleiche I/O-Ports).
ZVE2 **umgeht den Q240-Schutz** (privilegierter DMA-Zugriff).

### 6.1 Hardware-Modell

```
K2526-Karte
├── ZVE1 (Z80 cpu_)    – Haupt-CPU, Q240-Schutz aktiv für alle Zugriffe
├── ZVE2 (Z80 zve2_)   – DMA-CPU, Q240-Schutz bypassed (privelegiert)
│   ├── readByte  → bus_.memRead()    (direkt, kein Schutz)
│   ├── writeByte → bus_.memWrite()   (direkt, kein Schutz)
│   ├── readPort  → bus_.ioRead()     (gemeinsame I/O-Ports)
│   └── writePort → bus_.ioWrite()    (gemeinsame I/O-Ports)
└── BUSRQ/BUSAK-Protokoll:
    ─ K5122 assertiert /BUSRQ → ZVE1 gibt Bus frei
    ─ A5120Machine-Run-Loop führt ZVE2-Schritte aus (zve2Step())
    ─ zve2Step() gibt 0 zurück wenn /WAIT-ZVE2 aktiv (zve2_wait_=true)
    ─ K5122 gibt /BUSRQ frei, wenn ZVE2 letztes Byte gelesen hat
```

### 6.2 Reset-Steuerung (Port 04H, /RES-ZVE2)

Port 04H ist ein aktiv-low Reset-Signal für ZVE2:

| Wert | /RES-ZVE2 | Verhalten |
|------|-----------|-----------|
| Bit0 = 0 | assertiert | ZVE2 in Reset, führt keine Schritte aus |
| Bit0 = 1 | freigegeben | ZVE2 startet von PC=0000H |

```cpp
case 0x04:  // /RES-ZVE2 (active-low, bit 0)
    if (!(data & 0x01) && !zve2_reset_) {
        zve2_reset_ = true;         // → ZVE2 anhalten
    } else if ((data & 0x01) && zve2_reset_) {
        zve2_reset_ = false;
        zve2_.reset();              // → ZVE2 starten (PC=0000H)
    }
```

**Beim Systemstart** ist ZVE2 im Reset (Boot-ROM nutzt nur ZVE1 + dmaUpdate()-Fallback).
**Das Betriebssystem** kann ZVE2 freigeben (OUT 04H, 01H) und ein DMA-Programm im RAM
ausführen lassen, das Sektor-Daten über INIR/OTIR auf die K5122-Ports überträgt.

### 6.3 DMA-Ablauf mit ZVE2

```
ZVE1 schreibt /STR mit /WE=1 (Lesen):
  → K5122::handleCtrlPortAWrite()
      doReadSector()       ← Sektor sofort in sector_buf_ laden
      bus_.assertBUSRQ()

A5120Machine::run() – Run-Loop:
  if (bus_.isBUSRQ()) {
      if (!zre_.isZVE2InReset())
          zre_.zve2Step()          ← ZVE2 führt DMA-Programm aus (z.B. INIR)
      else
          afs_.dmaUpdate()         ← Fallback: sofortige Freigabe (Boot-ROM-Phase)
      remaining--;
      continue;
  }

ZVE2-DMA-Programm (Beispiel):
  LD BC, sector_size    ; Zähler
  LD HL, dest_addr      ; Zieladresse im RAM
  LD C, 16H             ; K5122 Daten-PIO Port B
  INIR                  ; BC Bytes von Port C → (HL), HL++, B--

K5122::ioRead(0x16) – letztes Byte:
  transferring_ = false
  bus_.releaseBUSRQ()    ← ZVE1 darf weiterarbeiten
```

### 6.4 SCHREIB-DMA mit ZVE2

```
ZVE1 schreibt /STR mit /WE=0 (Schreiben):
  → K5122::handleCtrlPortAWrite()
      sector_buf_.clear()    ← Buffer leeren für ZVE2-Eingabe
      bus_.assertBUSRQ()

ZVE2-DMA-Programm:
  LD BC, sector_size
  LD HL, src_addr           ; Quelladresse im RAM
  LD C, 14H                 ; K5122 Daten-PIO Port A
  OTIR                      ; BC Bytes von (HL) → Port C, HL++, B--
  ; Abschluss-Signal: erneut /STR schreiben (während Bus gehalten)
  OUT (10H), A              ; /STR-Flanke → K5122 erkennt ZVE2-Kontext

K5122::handleCtrlPortAWrite() – ZVE2-Kontext (bus_.isBUSRQ() true):
  doWriteSector()        ← sector_buf_ in Disk-Image schreiben
  bus_.releaseBUSRQ()
```

### 6.5 Fallback für Boot-ROM (ZVE2 im Reset)

Das 1 KB Boot-ROM verwendet **kein** ZVE2. Es liest Sektoren durch Byte-Polling
(`IN A,(16H)` Loop) nachdem der Bus freigegeben wurde. Der dmaUpdate()-Fallback
übernimmt in diesem Fall:

```
/STR-Flanke → doReadSector() füllt sector_buf_ → BUSRQ assertiert
dmaUpdate() → BUSRQ freigeben (sector_buf_ bleibt gefüllt)
ZVE1 (Boot-ROM) pollt Port 16H byte-weise → sector_buf_[buf_pos_++]
```

## 7. Speicher-/E/A-Schutz (Q 240) – implementiert

Die Q240-Schutzlogik unterteilt den 64 KB-Adressraum in **1024 Segmente à 64 Bytes**.
Jedes Segment hat ein Schutzbyte in der Tabelle (Bit 0 = Schreibschutz, Bit 1 = Leseschutz).

### 7.1 Klasse MemIOProtect

```cpp
class MemIOProtect {
public:
    static constexpr int SEGMENTS = 1024;

    void    enableTableWrite(bool active);  // /SPS-ESR: nächste ZVE1-Writes → Tabelle
    bool    isTableWriteEnabled() const;
    bool    isWriteProtected(uint16_t addr) const;  // Bit 0 des Segment-Eintrags
    bool    isReadProtected(uint16_t addr)  const;  // Bit 1 des Segment-Eintrags
    void    writeEntry(uint16_t addr, uint8_t flags);
    uint8_t readEntry(uint16_t addr)        const;
    void    setViolation();
    void    resetViolation();
    bool    isSPSViolation()                const;
    void    reset();

private:
    std::array<uint8_t, 1024> prot_table_{};
    bool table_write_enabled_ = false;
    bool sps_ind_             = false;
};
```

### 7.2 Schutz-Aktivierung durch /SPS-ESR (BS-PIO B2)

Wenn BS-PIO Port B Bit 2 (/SPS-ESR) auf 0 gesetzt wird, werden alle ZVE1-Schreibzugriffe
**in die Schutztabelle** umgeleitet statt ins RAM:

```
Segment-Nummer = Adresse >> 6           (Adresse / 64)
Tabellen-Eintrag = geliefertes Datenbyte
```

ZVE2 umgeht diesen Mechanismus vollständig (privilegierter DMA).

### 7.3 Schutz-Enforcement (ZVE1-Callbacks)

```cpp
// ZVE1 Lese-Callback
cpu_.readByte = [this](uint16_t a) -> uint8_t {
    if (spa_.isTableWriteEnabled()) return spa_.readEntry(a);
    if (spa_.isReadProtected(a))   { handleSPSViolation(); return 0xFF; }
    return bus_.memRead(a);
};

// ZVE1 Schreib-Callback
cpu_.writeByte = [this](uint16_t a, uint8_t d) {
    if (spa_.isTableWriteEnabled()) { spa_.writeEntry(a, d); return; }
    if (spa_.isWriteProtected(a))   { handleSPSViolation(); return; }
    bus_.memWrite(a, d);
};
```

### 7.4 Verletzungsbehandlung (handleSPSViolation)

Bei einem Schutzverstoß:
1. `sps_ind_` wird auf `true` gesetzt
2. BS-PIO Port A Bit 3 (SPS) wird auf 1 gesetzt (Input über `portAWrite`)
3. `bus_.assertNMI()` – ZVE1 erhält einen Nicht-Maskierbaren Interrupt

```cpp
void K2526::handleSPSViolation() {
    if (!sps_ind_) {
        sps_ind_ = true;
        spa_.setViolation();
        updatePortAInputs();
        bus_.assertNMI();
    }
}
```

### 7.5 Verletzung zurücksetzen (/RES-SPA, Port 02H)

Der CPU-Code quittiert einen Schutzverstoß durch Schreiben auf Port 02H:

```cpp
case 0x02:  // /RES-SPA
    spa_.resetViolation();
    sps_ind_      = false;
    int_bs_active_= false;
    updatePortAInputs();
    bus_.clearNMI();
    bs_pio_.portBWrite(buildPortBInputs());
    break;
```

### 7.6 /INT-BS (Port 00H) – OS-Ebenen-Wechsel-Interrupt

Schreiben auf Port 00H signalisiert einen gewünschten Betriebssystemebenen-Wechsel.
Die BS-PIO B1-Eingangsleitung (/INT-BS) wird dabei auf 0 gezogen:

```cpp
case 0x00:  // /INT-BS
    int_bs_active_ = true;
    bs_pio_.portBWrite(buildPortBInputs());
    break;
```

`buildPortBInputs()` setzt Bit 1 = 0, wenn `int_bs_active_` true ist:

```cpp
uint8_t K2526::buildPortBInputs() const {
    uint8_t val = (cfg_.config_bridges & 0x07) << 5;   // B5–B7: Brücken
    if (!int_bs_active_) val |= 0x02;                   // B1: /INT-BS (active-low)
    return val;
}
```

## 8. Boot-Sequenz

```
/RESET → ZVE1 startet bei 0000H
         ROM aktiv (1KB bei 0000H, read-only; Schreibzugriffe gehen an K3526-RAM)
         Q240 zurückgesetzt (alle Segmente ungeschützt)
         ROM-Code:
           1. RAM-Test (K3526)
           2. CTC initialisieren (Baud-Rate für SIO auf K8025)
           3. Ersten Sektor von Diskette lesen (K5122)
           4. Bootstrap-Code bei 0100H ausführen
           5. BS-PIO B0 auf 0 setzen → /LD-ROM aktiv → ROM abschalten
           6. Betriebssystem läuft nativ im RAM
```

Das Lade-ROM (1 KB, `doc/EPROMS/Prozessoreinheit_ZRE_K2525_A26.bin`) enthält den
oben beschriebenen Bootstrap-Code.
