## 1. Aufgabe

Die K5122 steuert bis zu 4 Diskettenlaufwerke. Sie ist keine direkte FDC-Implementierung wie der µPD765, sondern eine **PIO-basierte Lösung**: Zwei Z80-PIO-Chips übernehmen die Bit-by-Bit-Kommunikation mit den Laufwerken unter Steuerung der CPU.

Im Emulator wird die analoge Diskettenkommunikation durch **direkten Image-Zugriff** ersetzt.

## 2. Hardware-Zusammensetzung

```
K5122-Karte
├── Steuer-PIO A1.2 (Z80-PIO) – Ports 10H–13H
│   ├── Tor A Output: Steuersignale (/WE, MK, /STR, /ST, ...)
│   └── Tor B BitMode: Status-Ein-/Ausgänge (/RDYL, /TO, /WP, ...)
├── Daten-PIO A1.1 (Z80-PIO) – Ports 14H–17H
│   ├── Tor A Output: Schreibdaten (8 Bit)
│   └── Tor B Input: Lesedaten (8 Bit)
└── Laufwerksauswahl A4 (8212) – Port 18H
    └── /LCK1-4, /SE3, /SE4, /SET, /SEZ
```

## 4. Steuersignale (Steuer-PIO Tor A Ausgänge)

| Bit | Signal | Funktion |
|-----|--------|----------|
| 0 | /WE | Write Enable (Schreibkopf aktivieren) |
| 1 | MK | Mark — Feldwechsel-Strobe im **diskreten** 128-B-Lesemodell (IDAM→DATEN→nächster Sektor); steigende Flanke ⇒ `advanceField()` |
| 2 | /FR | Fault Reset — **ZUGLEICH die Seitenwahl**, am `/STR` gelatcht: **1 → Kopf 0, 0 → Kopf 1** |
| 3 | /STR | Start (Kommando starten, fallende Flanke = Strobe) |
| 4 | MK1 | Mark 1 — Re-Sync-Strobe im **kontinuierlichen** 1024-B-Lesemodell (3. Stufe); **fallende Flanke** (`0xB5→0x85`) ruft `resyncToNextMark()` und setzt den Strom auf die nächste Adressmarke (IDAM → DATEN → nächste IDAM), überspringt also die Datenbytes — modelliert das Re-Lock des Datenseparators |
| 5 | MR/SD | **Step-Richtung NUR** (am `/ST`-Puls gesampelt): **1 = inward (höhere Spur), 0 = outward (Richtung Spur 0)** |
| 6 | /HL | Head Load (Schreib-/Lesekopf aufsetzen) |
| 7 | /ST | Step (ein Schritt in Richtung MR/SD) |

> ⚠️ **Korrektur (verifiziert am 3rd-Stage-@OS.COM-Read, Commit `ab59ee8`)**: Die
> **Seitenwahl ist Bit 2 (/FR)**, NICHT Bit 5. Bit 5 (MR/SD) ist **ausschließlich die
> Step-Richtung** — es toggelt mit den MK/MK1-Re-Sync-Strobes und darf NIE als Seitenwahl
> interpretiert werden (sonst kippt der Kopf mitten im Transfer). `current_head_` wird daher
> **nur an der `/STR`-Flanke** aus Bit 2 gelatcht (`current_head_ = (data & 0x04) ? 0 : 1;`),
> nicht bei jedem Port-A-Write. Frühere Annahme (Bit 5 = Seitenwahl, Richtung 0=inward/1=outward)
> war doppelt falsch: head-1-Reads lieferten Kopf-0-Daten und der Read der zweiten Seite hing.

## 5. Status-Signale (Steuer-PIO Tor B Eingänge/Ausgänge)

| Bit | Signal | Richtung | Funktion |
|-----|--------|----------|----------|
| 0 | /RDYL | Eingang | Laufwerk bereit (MFS); 0 = bereit |
| 1 | MKE | Eingang | Marke erkannt |
| 2 | /HF | Ausgang | High Frequency (0=MFM 8", 1=FM 5") |
| 3 | PRE | Ausgang | Precompensation |
| 4 | /FA | Eingang | Fault Adapter |
| 5 | /WP | Eingang | Write Protect; 0 = schreibgeschützt |
| 6 | /FW | Eingang | **Fault** (Laufwerksfehler); 1 = kein Fehler |
| 7 | /TO | Eingang | **/TRACK 00** (Spur 00); 0 = Kopf auf Spur 0 |

> ⚠️ **Achtung – Bit 6/7 nicht verwechseln** (Quelle: `doc/trascripted/Floppy
> Anschlußsteuerung K 5122.md`, B₆=/FAULT=/FW, B₇=/TRACK 00=/TO). Das Boot-ROM
> wertet **Bit 7** (/TO = Spur 00) in der Laufwerkserkennung aus (ROM 0x0110:
> `IN (12H); RLCA; JR NC` → fortfahren, wenn Bit 7 = 0). `updateStatusPortB()`
> löscht daher Bit 7 bei Spur 0; Bit 6 (/FW) bleibt 1 (kein Fehler emuliert).

## 7. Transfer-Modell: ZVE2-DMA (BUSRQ/BUSAK)

Die K5122 überträgt Sektordaten über das BUSRQ/BUSAK-Protokoll. Beim realen
A5120-Boot führt **ZVE2** (zweite Z80-CPU auf der K2526) den Transfer aus; das
Boot-ROM startet ZVE2 aktiv (OUT 04H) und lässt es das STROBE_LOOP-Programm laufen.
Der frühere „Boot-ROM nutzt kein ZVE2"-Annahme ist überholt – Details und der
verifizierte Ablauf stehen in `doc/analyse_zre_rom_boot.md`.

### 7.1 Lesekopf über rotierender Spur (TrackImage-Streaming)

Der Datenpfad ist **formatagnostisch**: die K5122 kennt keine Sektorgrößen, Sektoranzahl,
CRC-Verfahren oder Boot-Stadien.  Bei einer Lese-`/STR`-Flanke baut `startReadTransfer()`
aus dem decodierten `TrackImage` des Laufwerks **on-the-fly** den Robotron-Spur-Layout-Stream
(`TrackCodec::buildRobotronTrack()`) in `robotron_track_` und setzt `cur_track_` darauf.
`ioRead(0x16)` streamt diese Bytes **byteweise** (`cur_track_->bytes[head_pos_]`) und läuft
am Spurende zyklisch um (die Diskette dreht weiter).  Es gibt **keine** controller-interne
Sektor-Synthese mehr — die ganze Struktur (Gaps, Sync, IDAM, DATEN, echte CRCs) steckt in
der Spur.

Robotron-Spurlayout pro Sektor (Marke liegt auf dem **A1**-Byte, nicht auf FE/FB):

```
[A1(Id)  FE cyl head sec size] [Gap 0x4E ×18/27] [A1(Data) FB <Daten> CRC CRC] [Gap 0x4E ×8]
```

Der Leser (ZVE2) synchronisiert seinen Datenseparator über die **MK (Bit 1)** bzw.
**MK1 (Bit 4)** Strobes: jede entsprechende Flanke ruft `resyncToNextMark()`, das `head_pos_`
auf die **nächste** Adressmarke im parallelen `marks[]`-Feld springen lässt (IDAM → DATEN →
nächste IDAM).  Das ist für 128-B- *und* 1024-B-Spuren **identisch** — der einzige Unterschied
ist, **wie viele Bytes** ZVE2 zwischen zwei Resyncs liest; der Controller weiß das nicht und
braucht keine `sector_size`-Fallunterscheidung.  (Der frühere `sector_buf_`/`buildField()`/
`stream_continuous_`-Ansatz mit größenabhängigen Sonderpfaden ist entfernt.)

### 7.2 Per-Byte-/BUSRQ + Bus-Verriegelung ZVE1↔ZVE2 (hardware-echt)

Laut K5122-Doku §5.6.1 entsteht `/BUSRQ` **pro Byte** aus dem RDY-Handshake des Daten-PIO
(„BUSRQ wird gebildet, wenn der Daten-PIO … bereit ist … durch /STR unterdrückt"): ZVE2
erhält den Bus, wenn ein Byte bereitliegt, holt es über Port 0x16 ab und verliert den Bus
wieder, bis das nächste Byte ~1 Byteperiode später bereitliegt.  In diesen Lücken läuft ZVE1.
Das ersetzt jede programm-/größenspezifische „wann ist die DMA fertig"-Sondererkennung.

```
K5122 (per-Byte-Drossel):
  /STR fallende Flanke (Port 10H Bit3, /WE=1) – ZVE1-Kontext:
    startReadTransfer(); byte_ready_=true; assertBUSRQ()
  ioRead(0x16):  Byte liefern; byte_ready_=false; releaseBUSRQ()      ← ZVE2 verliert Bus
  update(cycles): wenn !byte_ready_ und kBytePeriodCycles (~150) um:    ← nächstes Byte
                    byte_ready_=true; assertBUSRQ()
  update(cycles): /STR=1 ≥ kStrEndSampleCycles (~320) durchgehend →     ← Transfer-Ende
                    transferring_=false; releaseBUSRQ()                    (Latch-Abtastung)

A5120Machine::run() – Verriegelung:
  if (bus_.isBUSRQ() && ZVE2 aktiv) {
      zve2Step(); afs_.update(used2);   // während /BUSRQ läuft NUR ZVE2
      if ([0x03F8] 0→3 in dieser Runde) afs_.endDmaTransfer();   // ZVE2-Completion-Handshake
      continue;                          // KEIN nebenläufiges ZVE1
  }
```

**`/STR=1`-Latch-Abtastung (formatagnostisches Transfer-Ende).** Die Datenübertragung endet
laut Doku §5.5 „durch Abschalten von /STR".  Der Datenseparator tastet `/STR` aber nur am
Lesetakt ab (1 Byte ≈ 16 Bitzellen ≈ 150 Takte) — ein `/STR=1`-Puls kürzer als ~1 Byteperiode
wird **nicht** durchgetaktet (Latch).  Empirisch über einen ganzen Boot gemessen: Boot-ROM-
Setup-/Probe-Strobes halten `/STR=1` nur **11–18 Takte** (verschluckt), echte Track-Enden
**≥30000 Takte** (lösen aus).  Diese **eine** Regel ersetzt den früheren sektorgrößen-
abhängigen `OUT(13H),03H`-Track-Ende-Hack vollständig.

**ZVE2-Completion-Handshake `[0x03F8]=3`.** Zusätzlich erkennt `A5120Machine` den ZVE2-
Done-Flag (Boot-ROM `0x026B`; Sekundär- und 3.-Stufen-Lader nutzen denselben Flag) als
Transition 0→3 innerhalb der Runde und gibt dann den Bus frei.  Dies ist **kein** größen-
abhängiges Signal, sondern der programmweite „DMA fertig"-Handshake.  Er ist nötig, weil der
**Chained Loader (round 2)** die DMA per **Zeitschleife** abwartet (geladener Code `0x0441 =
DEC C; JR NZ`, kein Flag-Poll): unter der Per-Byte-Verschränkung verschiebt sich das
ZVE1:ZVE2-Tempo, sodass ZVE1s Zeitschleife **vor** ZVE2s DMA-Ende abläuft → `/STR=1` zu früh
→ unvollständiger Read → Retry-Schleife.  Der Flag-Watcher gibt den Bus exakt bei Completion
frei und umgeht diese Tempo-Abhängigkeit.  (Offen: ihn ganz durch ein reines HW-Signal zu
ersetzen — hängt am echten Arbitrierungsmodus, s. §7.6.)

> **Stand:** mit diesem Modell bootet die A5120 **vollständig** durch alle Loader-Stufen
> bis ins laufende CP/A-OS und erreicht die Uhrzeit-Eingabe-Aufforderung
> „Bitte Uhrzeit eingeben!" (Integrationstests `BootIntegration*`, `BootIntegrationCpa02.*`,
> `BootIntegrationDriveBC.*`).  Die größenabhängige Sonderbehandlung ist beseitigt.
>
> **Formaterkennung — Teil gelöst (2026-06-17, mit SAUBEREM Build verifiziert):** Die alte
> **Division durch 0** (`disks/cpadisk_02.img`, Boot ohne Uhr) ist **behoben**.  Ursache war
> die konkurrente ZVE1/ZVE2-Bus-Korruption — der ZVE2-Transfer lieferte nur Nullen, `SELDSK`
> baute einen Null-DPB (`[D1BE].spt=0`) und der Größenrechner dividierte durch 0 (`0xC7A3`).
> Mit der Per-Byte-Bus-Verriegelung liest ZVE2 die **echten** Sektordaten (`floppy_diag`: alle
> IDAMs FOUND, size=3) und `SELDSK` baut den DPB **korrekt** (`[D1BE]=0x28`=40 spt).
>
> *Noch offen:* `dir` **läuft trotzdem nicht bis zur Ausgabe** durch — beim eigentlichen
> Verzeichnis-Read dreht **ZVE2 in seiner Leseschleife (`0x2054–0x205C`)**, der Read schließt
> nicht ab (gleiche Read-Completion-Klasse wie der round-2-Lader / `@OS.COM`-3.-Stufe im
> laufenden OS).  *Korrektur:* eine frühere Notiz „dir listet das Verzeichnis bis A>" war ein
> **Stale-Object-Artefakt** (Held-Bus-Experiment-Reste im Build) — mit sauberem Build erscheint
> die dir-Ausgabe nicht.  **Build-Hygiene-Lehre:** nach Experimenten `build*/` neu konfigurieren,
> sonst täuschen alte Objektdateien Verhalten vor.

### 7.3 SCHREIB-DMA mit ZVE2

```
/STR fallende Flanke mit /WE=0 – ZVE1-Kontext:
  handleCtrlPortAWrite() → write_mode_=true; write_buf_.clear(); bus_.assertBUSRQ()

ZVE2-DMA-Programm: OTIR (Bytes (HL) → Port 14H, in write_buf_ gesammelt), dann
  erneuter /STR-Puls mit /WE=0.

handleCtrlPortAWrite() – /STR-Flanke bei gehaltenem BUSRQ + /WE=0:
  commitWrite()  ← write_buf_ in die gecachte Spur patchen (parseTrack→buildTrack)
  bus_.releaseBUSRQ()
```

### 7.4 Fallback: ZVE1-Byte-Polling (ZVE2 im Reset/WAIT)

Wenn ZVE2 im Reset oder durch /WAIT-ZVE2 angehalten ist (`isZVE2InReset()` bzw.
`isZVE2Waiting()`), übernimmt `dmaUpdate()` die Bus-Freigabe, und ZVE1 liest den
Spur-Stream selbst byteweise per `IN A,(16H)`. Dieser Pfad wird im realen
A5120-Boot **nicht** benutzt (dort läuft ZVE2), bleibt aber für OS-Szenarien und
Tests erhalten.

### 7.5 Laufwerksstatus (ctrl_pio_ Port B)

Nach jeder Statusänderung schreibt `updateStatusPortB()` das Statusbyte in den
Eingangslatch von ctrl_pio_; die CPU liest es über Port 0x12.

```
Kein Laufwerk: 0xF5 = /TO=1, /FW=1, /WP=1, /FA=1, /HF=1, PRE=0, MKE=0, /RDYL=1
Laufwerk eingelegt:  /RDYL=0 (bereit, Bit 0)
Auf Spur 0:          /TO=0   (Bit 7 = /TRACK 00; vom Boot-Laufwerkstest geprüft)
Schreibgeschützt:    /WP=0   (Bit 5)
/FW (Bit 6, Fault) bleibt 1: es wird kein Laufwerksfehler emuliert.
```

### 7.6 Bus-Arbitrierung: BUSRQ- vs WAIT-Betrieb (A5120 = gehaltener Bus)

Die K5122 kann laut Doku §5.6.1/§5.6.3 über **Brücken** in zwei Betriebsarten verdrahtet
werden (Decoder A3.3 aus den Daten-PIO-Quittungen `/ARDY`//`/BRDY`):

| Decoder-Ausgang | Bedingung | Wirkung |
|---|---|---|
| 00 | `/ARDY=0, /BRDY=0` | **`/WAIT`** — Warten auf das nächste Byte |
| 01 | `/ARDY=0, /BRDY=1` | **`/BUSRQ`** — Lesen: ein Byte vom PIO an den Bus |
| 02 | `/ARDY=1, /BRDY=0` | **`/BUSRQ`** — Schreiben: ein Byte vom Bus an den PIO |

**Welchen Modus nutzt die A5120?** Die K2526-Doku §6.7 (Z. 578–579) ist eindeutig:

> „Die 2. ZVE ist aktiv, **solange `/BUSRQ = low` bleibt**. Die Anschlußsteuerung des
> Floppy-Disk schaltet `/BUSRQ = 1`, **wenn der Datentransfer beendet ist**."

Also: **`/BUSRQ` bleibt für den GANZEN Sektor-/Spurtransfer low** — ZVE2 hält den Bus
durchgehend, ZVE1 ist die ganze Zeit **inaktiv** (Busse tristate).  `/BUSRQ` wird **erst am
Transfer-Ende** wieder high (Marken-FF-Reset durch MR oder Abschalten von `/STR`, §5.5).  Die
**Byte-für-Byte-Drosselung** innerhalb des gehaltenen Transfers erfolgt über das **interne
`/WAIT`** des Adapters (Decoder-Ausgang 00): ZVE2 wird zwischen den Bytes per `/WAIT`
angehalten, **behält dabei aber den Bus** und gibt ihn NICHT an ZVE1 zurück.

> ⚠️ Das `/WAIT-ZVE2` der ZRE (BS-PIO B3, §6.5) ist davon **getrennt**: es blendet nur einen
> `/WAIT`-Takt in die **`/M1`-Zyklen** der ZVE2 ein, damit ZVE2 ihr `/M1` nicht auf den
> Systembus gibt (sonst geriete die Interruptprioritätenkette durcheinander).  Es ist KEINE
> Daten-Byte-Drosselung.

**Konsequenz für unser Emulationsmodell (Stand 2026-06-17, ehrliche Einordnung):** Das aktuell
implementierte **Per-Byte-`/BUSRQ`** (§7.2) gibt den Bus **zwischen jedem Byte** an ZVE1 frei
(ZVE1 läuft in den Lücken).  Das ist eine **funktionierende Näherung** (voller Boot, alle Tests
grün), entspricht aber **nicht** exakt dem A5120-Modus „gehaltener Bus": dort läuft ZVE1
während des Transfers gar nicht, sondern nur (a) in der **Drehlatenz**, bevor `/BUSRQ`
assertiert (Daten noch nicht unter dem Kopf), und (b) **nach** dem Transfer-Ende.

Genau diese Abweichung erklärt die verbleibende `[0x03F8]=3`-Abhängigkeit (§7.2): weil unsere
Per-Byte-Lücken ZVE1 **während** der DMA laufen lassen, läuft die ZVE1-Zeitschleife des Chained
Loaders (`0x0441`) zu früh ab.  Im echten „gehaltener-Bus"-Modell stünde ZVE1 still und das
Tempo-Problem entfiele — die Freigabe käme rein aus „Datentransfer beendet" (Marken-FF-Reset /
`/STR`-Abschaltung durch ZVE2).  **Mögliche künftige Verfeinerung:** `/BUSRQ` für den ganzen
Transfer halten (Verriegelung ohne Per-Byte-Freigabe) + Drehlatenz vor dem ersten `/BUSRQ` +
Freigabe am ZVE2-seitigen Transfer-Ende — dann ließe sich der `[0x03F8]`-Watcher durch das
reine HW-Signal ersetzen.

**Verifiziert (2026-06-17): dieser Weg ist mit der vorhandenen Boot-Software NICHT trivial
gangbar — Sackgasse dokumentiert.** Ein Prototyp aus *gehaltenem Bus + Drehlatenz + ZVE2-Lese-
Inaktivitäts-Timeout als Transfer-Ende* wurde gebaut und scheiterte an drei ineinandergreifenden
Problemen:

1. **`[0x03F8]` funktioniert im gehaltenen Bus nicht allein.** Die Transition-Erkennung
   (`dma_saw_progress_`: warte bis ZVE1 `[0x03F8]=0` schreibt, dann akzeptiere `→3`) setzt
   voraus, dass ZVE1 **während** des Transfers mitläuft und den Flag löscht.  Im echten
   gehaltenen Bus ist ZVE1 jedoch eingefroren → der stale `=3` der Vorrunde wird nie gelöscht →
   Release feuert ab round 2 nie.  Genau die Nebenläufigkeit, die der gehaltene Bus beseitigen
   soll, ist Voraussetzung für den `[0x03F8]`-Handshake.

2. **Es gibt kein verlässliches HW-Transfer-Ende-Signal von ZVE2.** Die Boot-ROM zieht am
   ZVE2-Ende weder MR (bit5) noch `/STR=1`, sondern macht `OUT(11H,03)` + `[0x03F8]=3` und
   überlässt das spätere `/STR=1` dem (eingefrorenen!) ZVE1.  Das Ende musste daher per
   **Lese-Inaktivitäts-Timeout** geraten werden — wofür es **kein gutes Fenster** gibt: ~4000
   Takte feuern mitten in einer 1024-B-Lesung (No-Read-Lücke zwischen IDAM- und Datenfeld der
   3.-Stufen-Routine `0x1F7D`), ~50000 Takte feuern nie und fallen auf das (nach Punkt 1 nicht
   funktionierende) `[0x03F8]` zurück.  Resultat: Stage 0–2 grün, Stage 3 / running-OS rot.
   (Gotcha am Rande: ein atomares `INIR` macht 1024 `ioRead(0x16)` *innerhalb* eines
   `zve2Step()`, danach ruft der Run-Loop `afs_.update()` mit der vollen INIR-Zyklenzahl auf —
   ein Zähler-Reset im `ioRead` überschießt deshalb; nötig ist ein *Read-happened-Flag*, das
   `update()` auswertet.)

3. **Drehlatenz bricht den Sofort-`/BUSRQ`-Vertrag** der direkten K5122-Unit-Tests
   (`K2526ZVE2FloppyChain`, `WriteRoundtrip`), die `/BUSRQ` unmittelbar nach `/STR` erwarten.

**Schlussfolgerung:** Der gehaltene Bus ist das hardware-echte Modell (K2526 §6.7), aber die
A5120-Boot-/Loader-Software ist auf das `[0x03F8]`-Handshake **mit mitlaufendem ZVE1** ausgelegt;
strikt eingefrorenes ZVE1 ist damit inkompatibel.  Eine tragfähige Umsetzung bräuchte das exakte
HW-Transfer-Ende-Signal (die echte Marken-FF-Reset-Bedingung, vermutlich MR/bit5 oder das
Drainen des Daten-PIO) **byte-genau verifiziert** — solange das nicht vorliegt, bleibt das
**aktuelle Modell** (Per-Byte-`/BUSRQ` §7.2 + `[0x03F8]`-Handshake) der robuste Stand: voller
CP/A-Boot, Größenabhängigkeit beseitigt, alle Tests grün.

## 8. Laufwerksauswahl (8212, Port 18H)

```
Bit-Belegung Port 18H (Schreibzugriff):
  DO1 = /LCK2   DO5 = /LCK3
  DO2 = /LCK1   DO6 = /LCK4
  DO3 = /SE3    DO7 = /SET
  DO4 = /SE4    DO8 = /SEZ

Laufwerksauswahl durch:
  /LCK1 = 0 → Laufwerk A (drive 0)
  /LCK2 = 0 → Laufwerk B (drive 1)
  /LCK3 = 0 → Laufwerk C (drive 2)
  /LCK4 = 0 → Laufwerk D (drive 3)
```
